¿Qué es ABAP?

El lenguaje de SAP: cómo funciona y por qué es así

ABAP son las siglas de Advanced Business Application Programming. Es el lenguaje de programación propio de SAP. SAP es una empresa alemana que fabrica software de gestión empresarial (ERP) que usan miles de empresas en todo el mundo. ABAP es el lenguaje que permite personalizar y crear programas dentro de ese entorno.

Imagina que SAP es un edificio enorme con miles de habitaciones ya construidas (transacciones estándar). ABAP es el lenguaje que te permite reformar una habitación, construir una nueva o conectar varias habitaciones para crear un flujo de trabajo a medida. No puedes cambiar los cimientos del edificio, pero sí adaptar el espacio a las necesidades de tu empresa.

¿Dónde se ejecuta ABAP?

ABAP no se ejecuta en tu ordenador personal. Tú escribes el código en el editor de SAP (usando la aplicación SAP GUI, la que tienes instalada con versión 8000.1.0.1161), pero el programa se guarda y se ejecuta en un servidor SAP central. Cuando ejecutas un programa, el servidor lo procesa y te envía el resultado a tu pantalla. Esto es muy diferente a lenguajes como Python o JavaScript, que se ejecutan localmente.

El punto y final: el separador de sentencias

En ABAP, cada sentencia termina con un punto . (punto y final). Es como el punto y coma ; en otros lenguajes. Si olvidas el punto, SAP te dará un error porque no sabe dónde acaba la instrucción.

Ejemplo mínimo
WRITE 'Hola mundo'.   " El punto indica que aquí termina la orden
🧠
Los comentarios

Todo lo que va después de " (doble comilla) en una línea es un comentario. SAP lo ignora completamente. Sirve para explicar qué hace el código.

La estructura de un programa REPORT

Los bloques que componen cualquier programa ABAP

Los programas que has escrito son de tipo REPORT. Son los más comunes. Tienen una estructura fija, aunque no todos los bloques son obligatorios. Vamos a ver el esqueleto típico y luego cada pieza por separado.

Estructura básica de un REPORT
REPORT zejercicio_xx.

" 1. Declaraciones globales (TYPES, DATA, TABLES, CONSTANTS...)
TYPES: ...
DATA:  ...
TABLES: ...

" 2. Pantalla de selección (opcional)
SELECTION-SCREEN BEGIN OF BLOCK ...
  PARAMETERS: ...
  SELECT-OPTIONS: ...
SELECTION-SCREEN END OF BLOCK ...

" 3. Evento INITIALIZATION (opcional, se ejecuta antes de mostrar la pantalla)
INITIALIZATION.
  ...

" 4. Evento AT SELECTION-SCREEN (opcional, reacciona a acciones del usuario)
AT SELECTION-SCREEN.
  ...

" 5. Evento AT SELECTION-SCREEN OUTPUT (opcional, se ejecuta antes de dibujar la pantalla)
AT SELECTION-SCREEN OUTPUT.
  ...

" 6. Evento START-OF-SELECTION (el "main", se ejecuta al pulsar Ejecutar)
START-OF-SELECTION.
  ...

" 7. Subrutinas (FORM) al final (opcional)
FORM ...
ENDFORM.

Todo lo que está fuera de los eventos (como TYPES, DATA) se declara al principio y es global: cualquier evento o subrutina puede ver esas variables. Lo que se declara dentro de un evento o subrutina es local y solo existe ahí.

⚠️
Importante: Los eventos no se llaman unos a otros. SAP los llama en el orden adecuado según lo que haga el usuario. Por eso no ves una función main() que lo controle todo.
📛

Convenciones de nombres (los prefijos)

Cómo saber de un vistazo qué tipo de variable es

En ABAP es muy común usar prefijos de 2-3 letras para indicar el tipo y el ámbito de una variable. No es obligatorio, pero el 99% de los programadores lo hacen porque hace el código muchísimo más legible.

Por ejemplo, si ves ls_producto en el código, inmediatamente sabes que es una estructura local (una sola fila de datos). No necesitas buscar su declaración para saberlo.

Tabla completa de prefijos (con ejemplos de tus programas)

PrefijoSignificadoQué contieneEjemplo
ty_ Type (molde) Una definición de tipo, no contiene datos. Es un plano. ty_producto, ty_tarea
tt_ Table Type (tipo tabla) El tipo de una tabla interna (lista de filas). tt_productos, tt_numeros
ls_ Local Structure Una estructura (una fila) que solo existe en el ámbito local (dentro de un evento o FORM). ls_cat, ls_tarea
lt_ Local Table Una tabla interna local. lt_catalogo, lt_productos
gs_ Global Structure Estructura declarada al principio del programa, accesible desde cualquier parte. gs_coleccionEmpleado
gt_ Global Table Tabla interna global. gt_tablaInternaempleados
gr_ Global Reference Referencia a un objeto (OOP). gr_alv, gr_dock
lv_ Local Variable Variable simple local (número, texto, fecha). lv_contador
p_ Parameter Parámetro de pantalla de selección. p_lista, p_cmpn
<fs_> Field Symbol Puntero a datos. No guarda datos, apunta a ellos. <fs_tarea>
✅
Consejo: Acostúmbrate a usar estos prefijos desde el principio. Cuando tu programa tenga 500 líneas, te alegrarás de haberlo hecho.

TYPES — Los moldes de datos

Definir la forma de los datos sin ocupar memoria

TYPES sirve para definir un tipo de datos. Piensa en él como un molde para hacer galletas: defines la forma (redonda, con borde ondulado), pero aún no has hecho ninguna galleta. No ocupa memoria, solo describe cómo serán los datos cuando los crees.

Estructura: BEGIN OF … END OF

Cuando necesitas agrupar varios campos relacionados (como los datos de un producto: ID, nombre, precio), defines una estructura con BEGIN OF y END OF.

zejercicio_5_sq_botones.abap
" Definimos el molde para un producto
TYPES: BEGIN OF ty_producto,   " ← nombre del molde
         id     TYPE string,     " ← campo id de tipo texto
         nombre TYPE string,     " ← campo nombre de tipo texto
       END OF ty_producto.    " ← fin del molde

" Ahora ty_producto es un tipo que tiene dos campos: id y nombre.
" Todavía no hay ninguna variable en memoria que use este molde.

Tipos de datos básicos

Dentro de una estructura, cada campo tiene un tipo. Los más comunes son:

TipoNombreQué guardaValor inicial
stringCadena de longitud variableTexto de cualquier tamañovacío ( '')
iEnteroNúmeros enteros (sin decimales)0
p DECIMALS nEmpaquetadoNúmeros con n decimales exactos0
char10Texto de longitud fija 10Texto de exactamente 10 caracteres (se rellena con espacios)10 espacios
dFechaFecha en formato AAAAMMDD00000000

Tipo tabla: TABLE OF

Una vez que tienes el molde de una fila, puedes definir el molde de una lista de esas filas: una tabla interna.

Tipo tabla
TYPES: tt_productos TYPE TABLE OF ty_producto WITH EMPTY KEY.
"   ↑                   ↑                   ↑                ↑
" nombre del tipo    "tabla de"         el tipo de cada fila   "sin clave" (ABAP moderno)

" Ahora tt_productos es un tipo que representa una lista de productos.
⚠️
WITH EMPTY KEY es una exigencia de ABAP moderno (desde 7.40). Le dices a SAP que esta tabla no tiene una clave primaria especial, es solo una lista. No te preocupes por ello ahora.
📦

DATA — Crear variables (ocupar memoria)

Aquí es cuando realmente reservas espacio para guardar datos

Si TYPES es el plano de un edificio, DATA es construir el edificio. Cuando declaras una variable con DATA, SAP reserva un espacio en la memoria del servidor para que puedas guardar valores.

Toda variable tiene un valor inicial por defecto: los textos empiezan vacíos, los números en 0, las tablas sin filas, las referencias a objetos en "null" (INITIAL).

Formas de declarar DATA

Todos los patrones que has usado
" 1. Estructura (una fila) basada en un tipo definido con TYPES
DATA: ls_cat TYPE ty_producto.
"     ↑      ↑
"   nombre   "del tipo ty_producto"
" Ahora ls_cat tiene dos campos: ls_cat-id y ls_cat-nombre, ambos vacíos.

" 2. Tabla interna (lista) basada en un tipo tabla
DATA: lt_catalogo TYPE tt_productos.
" Es una tabla vacía (0 filas).

" 3. Variable simple con valor inicial
DATA: lv_contador TYPE i VALUE 0.
" VALUE 0 es opcional porque los números ya empiezan en 0.

" 4. Referencia a un objeto (OOP)
DATA: gr_alv TYPE REF TO cl_salv_table.
" gr_alv no apunta a ningún objeto todavía. Está INITIAL.

" 5. Estructura de trabajo para una tabla del diccionario (necesario para SELECT-OPTIONS)
TABLES: sflight.
" Esto declara una estructura llamada sflight con los mismos campos que la tabla de base de datos sflight.
" No es una tabla interna, es una estructura de una sola fila.

Cómo acceder a los campos de una estructura

Para leer o escribir un campo concreto de una estructura, usas el guión -. No es un signo menos, es el separador de campos en ABAP.

Acceso a campos
ls_cat-id     = '1'.        " Asigna '1' al campo id de la estructura ls_cat
ls_cat-nombre = 'Laptop'.   " Asigna 'Laptop' al campo nombre
WRITE: ls_cat-nombre.       " Muestra 'Laptop'

Conversión implícita de tipos

Cuando pasas un literal como '2.0' a un parámetro que espera TYPE p (número empaquetado), ABAP intenta convertirlo automáticamente. Esto funciona porque el sistema interpreta el texto como número. Sin embargo, es más claro y seguro usar literales numéricos directamente: 2 o 2.0.

zejercicio_02_sq_dia7.abap
" Esto funciona (convierte '2.0' a número)
go_calc->set_valores( i_val1 = '2.0' i_val2 = '2.0' ).

" Pero esto es más correcto
go_calc->set_valores( i_val1 = 2 i_val2 = 2 ).

Recomendación: Siempre que sea posible, pasa el tipo de dato exacto que espera el método. Evita depender de conversiones implícitas, aunque ABAP sea muy permisivo.

🚨
¡Ojo! El punto y coma no existe en ABAP. Las instrucciones terminan con punto. No pongas punto y coma después de cada línea o tendrás errores.
🖥️

PARAMETERS — Campos de entrada en pantalla

La forma más simple de pedir un dato al usuario

PARAMETERS crea automáticamente un campo de entrada en la pantalla de selección (la pantalla gris que aparece antes de que el programa se ejecute). El usuario escribe un valor y, al pulsar Ejecutar, ese valor se guarda en la variable que declaraste.

SAP se encarga de todo: dibuja el campo, le asigna una etiqueta (el nombre de la variable, pero puedes cambiarlo con elementos de texto), y si el tipo de dato lo permite, añade ayuda de búsqueda (F4).

Ejemplos de PARAMETERS
" Parámetro simple de tipo texto (char10)
PARAMETERS: p_nombre TYPE char10.

" Parámetro que hereda el tipo de un campo de base de datos (del diccionario SAP)
PARAMETERS: p_carrid TYPE sflight-carrid.
" Esto hace que el campo tenga la misma longitud, validaciones y ayuda F4 que el campo carrid de la tabla sflight.

" Parámetro que se convierte en un desplegable (LISTBOX)
PARAMETERS: p_lista TYPE char10 AS LISTBOX VISIBLE LENGTH 40.
" AS LISTBOX: dibuja un combo box en lugar de un campo de texto.
" VISIBLE LENGTH 40: ancho en caracteres del desplegable.

" Radiobuttons (botones de opción)
PARAMETERS: rb_opc1 RADIOBUTTON GROUP grp1.
PARAMETERS: rb_opc2 RADIOBUTTON GROUP grp1.
" Ambos pertenecen al grupo 'grp1'. Solo uno puede estar marcado a la vez.
💡
¿Qué es sflight-carrid?

sflight es una tabla de base de datos que SAP proporciona para ejemplos (vuelos). carrid es un campo de esa tabla (identificador de aerolínea). Al escribir TYPE sflight-carrid le estás diciendo a SAP: "crea un campo con el mismo tipo que tiene ese campo en la base de datos". Así te ahorras tener que recordar si es char 3 o char 10.

🎛️

SELECT-OPTIONS — Rangos de selección

Cuando un solo valor no basta: permite rangos, exclusiones y múltiples valores

Un SELECT-OPTIONS es como un PARAMETERS pero mucho más potente. El usuario puede introducir:

  • Un valor único: 100
  • Un rango: 100 a 200
  • Una lista: 100, 150, 200
  • Exclusiones: distinto de 150
  • Patrones: 1* (todo lo que empiece por 1)

En la pantalla, aparece un campo con un botón a la derecha. Al pulsarlo, se abre una ventana donde el usuario puede añadir todas las condiciones que quiera.

Declaración de SELECT-OPTIONS
TABLES: sflight.  " Obligatorio para referenciar campos de esa tabla

SELECT-OPTIONS: so_conn FOR sflight-connid.
"                ↑              ↑
"           nombre de la   campo de referencia
"           variable

Internamente, so_conn se convierte en una tabla con cuatro columnas. Cada fila representa una condición. La tabla tiene este aspecto:

CampoSignificadoValores ejemplo
SIGNIncluir (I) o Excluir (E)'I' para incluir, 'E' para excluir
OPTIONOperador'EQ' (igual), 'BT' (entre), 'CP' (contiene patrón), 'GT' (mayor que)...
LOWValor inferior (o único)'100'
HIGHValor superior (solo para rangos)'200'

Cuando luego uses esa variable en una sentencia SQL (por ejemplo, SELECT ... WHERE connid IN so_conn), SAP evaluará automáticamente todas las condiciones que el usuario introdujo.

🎯
Ejemplo de uso en START-OF-SELECTION: 
SELECT * FROM sflight INTO TABLE lt_vuelos WHERE connid IN so_conn.
SAP hará la consulta teniendo en cuenta todos los rangos y exclusiones que puso el usuario. Tú no tienes que escribir ningún IF.
🔘

PUSHBUTTON — Botones personalizados

Añadir botones propios a la pantalla de selección

Por defecto, la pantalla de selección solo tiene los botones estándar de SAP: Ejecutar (F8), Cancelar, etc. Con PUSHBUTTON puedes añadir tus propios botones, como "Añadir", "Eliminar" o "Calcular".

Cada botón tiene un USER-COMMAND, que es un código (hasta 20 caracteres) que SAP guarda en la variable del sistema sy-ucomm cuando el usuario pulsa el botón. Así puedes saber qué botón se pulsó.

Paso a paso

1
Declarar el botón en la pantalla de selección

Usas SELECTION-SCREEN PUSHBUTTON. Le das una posición (fila/columna), un nombre de variable que contendrá el texto del botón, y un USER-COMMAND.

2
Asignar el texto visible en INITIALIZATION

En el evento INITIALIZATION, asignas a esa variable el texto que quieres que muestre el botón (por ejemplo, 'Añadir').

3
Reaccionar en AT SELECTION-SCREEN

En AT SELECTION-SCREEN, compruebas el valor de sy-ucomm para saber qué botón se pulsó y ejecutar la acción correspondiente.

zejercicio_5_sq_botones.abap
" 1. Declaración (arriba, con los DATA)
DATA: btn_add TYPE char10,   " texto del botón Añadir
      btn_fin TYPE char10.   " texto del botón Finalizar

SELECTION-SCREEN:
  PUSHBUTTON /1(10) btn_add USER-COMMAND add,
  PUSHBUTTON 25(10) btn_fin USER-COMMAND fin.

" 2. En INITIALIZATION asignamos los textos
INITIALIZATION.
  btn_add = 'Añadir'.
  btn_fin = 'Finalizar'.

" 3. En AT SELECTION-SCREEN detectamos la pulsación
AT SELECTION-SCREEN.
  CASE sy-ucomm.
    WHEN 'ADD'.
      " Lógica para añadir producto
    WHEN 'FIN'.
      " Lógica para finalizar pedido
  ENDCASE.
🚨
Mayúsculas en USER-COMMAND: SAP convierte automáticamente el comando a mayúsculas. Si en el WHEN pones 'Add' no coincidirá. Pon siempre 'ADD'.
📋

LISTBOX — Desplegables dinámicos

Cómo llenar un combo box con valores de tu programa

Para crear un desplegable (combo box) en la pantalla de selección, necesitas dos cosas:

  1. Declarar un PARAMETERS con la adición AS LISTBOX.
  2. Llenar una tabla especial llamada vrm_values y pasársela al módulo de función VRM_SET_VALUES en INITIALIZATION.

vrm_values es un tipo de datos de SAP que tiene dos campos:

  • key: el valor interno que recibirá la variable del parámetro.
  • text: el texto que verá el usuario en el desplegable.
Flujo completo de LISTBOX
" 1. Declaración del parámetro
PARAMETERS: p_lista TYPE char10 AS LISTBOX VISIBLE LENGTH 40.

" 2. En INITIALIZATION, preparamos las opciones
INITIALIZATION.
  DATA: lt_valores TYPE vrm_values,   " tabla de opciones
        ls_valor   TYPE vrm_value.   " fila de la tabla

  " Supongamos que tenemos una tabla lt_catalogo con los productos
  LOOP AT lt_catalogo INTO ls_cat.
    ls_valor-key  = ls_cat-id.      " el ID que guardaremos
    ls_valor-text = ls_cat-nombre.   " el texto visible
    APPEND ls_valor TO lt_valores.
  ENDLOOP.

  " 3. Llamamos al módulo de función que conecta la tabla con el campo
  CALL FUNCTION 'VRM_SET_VALUES'
    EXPORTING
      id     = 'P_LISTA'      " nombre del PARAMETERS en MAYÚSCULAS
      values = lt_valores.
🚨
Error mortal: El parámetro id de VRM_SET_VALUES debe ir en mayúsculas. Si pones 'p_lista' en minúsculas, el desplegable aparecerá vacío y no sabrás por qué.
🔵

RADIOBUTTON GROUP — Botones de opción

Elegir una opción entre varias

Los radiobuttons se declaran con PARAMETERS y la adición RADIOBUTTON GROUP. Todos los que compartan el mismo grupo pertenecen al mismo conjunto: solo uno puede estar marcado a la vez.

Cada radiobutton es una variable de tipo char1. Cuando está marcado, su valor es 'X'. Cuando no, es un espacio (' ').

Radiobuttons
PARAMETERS: rb_bas RADIOBUTTON GROUP mod DEFAULT 'X',
            rb_fin RADIOBUTTON GROUP mod,
            rb_com RADIOBUTTON GROUP mod.

" En START-OF-SELECTION podemos preguntar:
IF rb_bas = 'X'.
  " Modo básico activado
ELSEIF rb_fin = 'X'.
  " Modo financiero
ELSE.
  " Modo completo
ENDIF.
⚡

USER-COMMAND en radiobuttons

Forzar la actualización de la pantalla al cambiar de opción

Cuando el usuario cambia de radiobutton, la pantalla no se redibuja automáticamente. Para que los cambios (como activar/desactivar campos) se vean al instante, necesitas que al pulsar el radiobutton se dispare un evento de pantalla. Eso se consigue añadiendo USER-COMMAND al grupo.

zejercicio_15_sq_dia6.abap
PARAMETERS: rb_bas RADIOBUTTON GROUP mod USER-COMMAND click DEFAULT 'X',
            rb_fin RADIOBUTTON GROUP mod,
            rb_com RADIOBUTTON GROUP mod.

Con USER-COMMAND click, cada vez que el usuario selecciona un radiobutton del grupo, SAP ejecuta AT SELECTION-SCREEN (con sy-ucomm = 'CLICK') y luego AT SELECTION-SCREEN OUTPUT, lo que nos permite reevaluar la lógica de activación de campos.

🔄

LOOP AT SCREEN — Modificar la pantalla dinámicamente

Activar, desactivar u ocultar campos según el contexto

En ABAP, la pantalla de selección se describe internamente mediante una tabla llamada SCREEN. Cada fila de esta tabla representa un elemento de la pantalla (un campo, un texto, un botón) y tiene atributos como:

  • screen-input: 1 si el campo es modificable, 0 si es de solo lectura.
  • screen-invisible: 1 si el campo está oculto.
  • screen-group1: un grupo de modificación que asignamos al declarar el campo (con MODIF ID).

En el evento AT SELECTION-SCREEN OUTPUT (que se ejecuta justo antes de mostrar la pantalla), podemos recorrer esta tabla y cambiar los atributos para adaptar la pantalla a lo que el usuario ha elegido.

Paso 1: Asignar grupos de modificación a los campos

Al declarar un PARAMETERS o SELECT-OPTIONS, puedes añadir MODIF ID grupo. El grupo es una etiqueta de hasta 3 caracteres que luego usaremos en el LOOP.

Asignar grupos
PARAMETERS: p_noper TYPE string MODIF ID bas,
            p_falta TYPE d      MODIF ID fec,
            p_bank  AS CHECKBOX MODIF ID fin.

Paso 2: En AT SELECTION-SCREEN OUTPUT modificar SCREEN

Lógica de activación
AT SELECTION-SCREEN OUTPUT.
  LOOP AT SCREEN.                        " recorremos todos los elementos
    CASE screen-group1.                  " según el grupo...
      WHEN 'BAS'.
        IF rb_bas = 'X' OR rb_com = 'X'.
          screen-input = 1.               " activado
        ELSE.
          screen-input = 0.               " desactivado (solo lectura)
        ENDIF.
      WHEN 'FEC'.
        IF rb_fin = 'X'.
          screen-input = 0.               " desactivado en modo financiero
        ELSE.
          screen-input = 1.
        ENDIF.
    ENDCASE.
    MODIFY SCREEN.                       " ¡Obligatorio! Guarda los cambios
  ENDLOOP.
⚠️
No olvides MODIFY SCREEN. Si no lo pones, los cambios que hiciste en la línea de SCREEN se pierden.
🗂️

BLOCK WITH FRAME TITLE — Agrupar campos visualmente

Recuadros con título para organizar la pantalla

Cuando tienes muchos campos, puedes agruparlos dentro de un recuadro con título. Esto mejora la usabilidad.

zejercicio_sq_panelseleccion.abap
SELECTION-SCREEN BEGIN OF BLOCK b1 WITH FRAME TITLE TEXT-001.
  PARAMETERS: p_carrid TYPE sflight-carrid.
  SELECT-OPTIONS: so_conn FOR sflight-connid.
SELECTION-SCREEN END OF BLOCK b1.

TEXT-001 es un elemento de texto. Puedes definirlo en Goto → Text Elements → Text Symbols del editor ABAP. Así el título puede traducirse a otros idiomas.

Eventos del programa — El ciclo de vida

Por qué el código no se ejecuta de arriba a abajo

En ABAP, el programa no se ejecuta en orden secuencial como en Python. En lugar de eso, tú escribes bloques de código asociados a eventos. SAP decide cuándo llamar a cada bloque según lo que haga el usuario.

Los eventos principales son:

Evento 1

INITIALIZATION

Se ejecuta una sola vez, antes de que la pantalla de selección aparezca. Aquí preparas datos iniciales: cargar catálogos, asignar textos a botones, rellenar listboxes.

Evento 2

AT SELECTION-SCREEN

Se ejecuta cada vez que el usuario interactúa con la pantalla: pulsa un botón, presiona Enter, o hace clic en Ejecutar. Usas sy-ucomm para saber qué acción ocurrió.

Evento 3

AT SELECTION-SCREEN OUTPUT

Se ejecuta justo antes de que la pantalla se dibuje (al inicio y después de cada interacción que requiera redibujado). Aquí va la lógica para activar/desactivar campos.

Evento 4

START-OF-SELECTION

Se ejecuta cuando el usuario pulsa Ejecutar (F8) y la pantalla de selección ha pasado todas las validaciones. Aquí va la lógica principal del programa.

💡
¿Por qué no puedo poner código suelto?

Todo el código ejecutable debe estar dentro de un evento o de una subrutina (FORM). Si pones instrucciones fuera de estos bloques, SAP te dará error.

🔮

Variables del sistema sy-

El salpicadero de SAP: información que SAP actualiza automáticamente

Las variables que empiezan por sy- (o syst-) son variables globales de solo lectura que SAP mantiene actualizadas en todo momento. No las declaras, ya existen. Contienen información sobre el estado del programa, del sistema y de la última operación.

Son como los indicadores del coche: no puedes cambiarlos, pero te dicen la velocidad, las revoluciones, si hay algún error, etc.

Las más importantes (y las que has usado)

VariableQué contieneCuándo se actualizaEjemplo de uso
sy-subrc Código de retorno de la última operación. Después de cada instrucción que puede fallar (READ TABLE, SELECT, CALL FUNCTION). 
READ TABLE lt_cat INTO ls_cat WITH KEY id = p_id.
IF sy-subrc = 0.
  " encontrado
ENDIF.
sy-ucomm El USER-COMMAND de la última acción del usuario. En AT SELECTION-SCREEN, después de que el usuario pulse un botón o Enter. 
CASE sy-ucomm.
  WHEN 'ADD'.
    " añadir
ENDCASE.
sy-index Número de iteración actual en un bucle DO o LOOP (empieza en 1). En cada vuelta de un DO o LOOP. 
DO 10 TIMES.
  WRITE: / sy-index.
ENDDO.
sy-tabix Índice de la fila actual en un LOOP AT (empieza en 1). En cada vuelta de LOOP AT. 
LOOP AT lt_tab INTO ls_fila.
  WRITE: / sy-tabix, ls_fila.
ENDLOOP.
sy-uname Nombre de usuario SAP que está ejecutando el programa. Siempre disponible. 
ls_log-usuario = sy-uname.
sy-datum Fecha actual del sistema en formato AAAAMMDD. Siempre disponible. 
IF ls_fecha < sy-datum.
  " fecha pasada
ENDIF.
sy-uzeit Hora actual en formato HHMMSS. Siempre disponible. 
ls_log-hora = sy-uzeit.
🚨
Nunca asignes valores a sy- (salvo rarísimas excepciones). Son de solo lectura. Si lo haces, puedes desestabilizar el programa.

sy-subrc en detalle

Es la variable más usada. Después de cada operación que puede fallar, SAP pone un número en sy-subrc. Los valores típicos son:

  • 0: éxito, encontró algo, la operación fue bien.
  • 4: no encontró nada, fin de datos, etc.
  • 8, 12, etc.: otros errores específicos de cada función.

La regla de oro: comprueba sy-subrc inmediatamente después de la operación. Si haces otra cosa entre medias, su valor puede cambiar.

Uso correcto de sy-subrc
READ TABLE lt_catalogo INTO ls_cat WITH KEY id = p_lista.
IF sy-subrc = 0.
  " Encontrado, podemos usar ls_cat
ELSE.
  " No encontrado, mostrar mensaje de error
ENDIF.

IF / ELSEIF / ELSE — Tomar decisiones

Ejecutar código solo si se cumple una condición

El IF funciona igual que en cualquier lenguaje. La diferencia está en algunos operadores específicos de ABAP.

Operadores de comparación

  • = o EQ: igual que
  • <> o NE: distinto de
  • > o GT: mayor que
  • < o LT: menor que
  • >= o GE: mayor o igual
  • <= o LE: menor o igual

Comprobaciones especiales

  • IS INITIAL: ¿la variable tiene su valor por defecto? (texto vacío, número 0, tabla sin filas, objeto sin crear).
  • IS NOT INITIAL: lo contrario.
Ejemplos
IF p_lista IS INITIAL.
  MESSAGE 'Debe seleccionar un producto' TYPE 'E'.
ENDIF.

IF lt_tabla IS NOT INITIAL.
  WRITE: / 'La tabla tiene', LINES( lt_tabla ), 'filas'.
ENDIF.

IF nota >= 5 AND nota <= 10.
  WRITE: 'Aprobado'.
ELSEIF nota < 5.
  WRITE: 'Suspenso'.
ELSE.
  WRITE: 'Nota inválida'.
ENDIF.
💡
IS INITIAL vs = '': IS INITIAL funciona para cualquier tipo de datos. = '' solo para cadenas. Mejor usar IS INITIAL siempre.
🃏

CASE / WHEN — Múltiples caminos

Alternativa a muchos ELSEIF sobre la misma variable

Cuando tienes que comparar una misma variable con muchos valores, CASE es más limpio que una cadena de ELSEIF.

CASE típico con sy-ucomm
CASE sy-ucomm.
  WHEN 'ADD'.
    PERFORM add_product.
  WHEN 'FIN'.
    PERFORM finish_order.
  WHEN 'DEL' OR 'REMOVE'.
    PERFORM delete_item.
  WHEN OTHERS.
    " No hacer nada o mensaje
ENDCASE.

LOOP AT — Recorrer tablas internas

La forma de procesar cada fila de una lista

Hay dos formas principales de recorrer una tabla interna: INTO (copia) y ASSIGNING (puntero).

LOOP AT … INTO (copia)

En cada iteración, SAP copia la fila actual a una variable estructura. Si modificas esa variable, la tabla original no cambia. Para actualizar la tabla, necesitas un MODIFY explícito.

LOOP INTO
LOOP AT lt_productos INTO ls_producto.
  WRITE: / ls_producto-nombre.
  ls_producto-precio = ls_producto-precio * 1.1.  " Esto NO cambia lt_productos
ENDLOOP.

LOOP AT … ASSIGNING (puntero)

Usamos un field symbol. En cada iteración, el field symbol apunta directamente a la fila en memoria. Cualquier cambio en el field symbol modifica la tabla original.

LOOP ASSIGNING
FIELD-SYMBOLS: <fs_prod> TYPE ty_producto.

LOOP AT lt_productos ASSIGNING <fs_prod>.
  <fs_prod>-precio = <fs_prod>-precio * 1.1.  " Esto SÍ modifica lt_productos
ENDLOOP.
✅
Cuándo usar cada uno: Usa INTO si solo necesitas leer los datos. Usa ASSIGNING si vas a modificar la tabla.

Bucle sin tabla, solo contador

DO n TIMES ejecuta el bloque exactamente n veces. En cada vuelta, la variable del sistema sy-index contiene el número de iteración (empezando en 1).

zejercicio_sq_15_dia4.abap

  ls_numero-numero = sy-index.
  APPEND ls_numero TO lt_numeros.
ENDDO.

Tablas internas — Listas en memoria

El corazón de cualquier programa ABAP

Una tabla interna es simplemente una lista de filas que viven en la memoria del servidor mientras el programa se ejecuta. No se guardan en base de datos. Puedes añadir filas, quitarlas, ordenarlas, buscarlas...

Patrón de 3 pasos

1
Definir el tipo de la fila (TYPES)

Crear un molde con TYPES BEGIN OF.

2
Declarar la tabla y una estructura de trabajo (DATA)

DATA: lt_tabla TYPE TABLE OF ty_fila, ls_fila TYPE ty_fila.

3
Operar con la tabla

APPEND, READ, LOOP, DELETE, CLEAR, etc.

Ejemplo completo
TYPES: BEGIN OF ty_empleado,
         id     TYPE i,
         nombre TYPE string,
       END OF ty_empleado.

DATA: lt_empleados TYPE TABLE OF ty_empleado,
      ls_empleado  TYPE ty_empleado.

ls_empleado-id = 1. ls_empleado-nombre = 'Ana'.
APPEND ls_empleado TO lt_empleados.

CLEAR ls_empleado.   " Limpiamos para la siguiente fila
ls_empleado-id = 2. ls_empleado-nombre = 'Luis'.
APPEND ls_empleado TO lt_empleados.

LOOP AT lt_empleados INTO ls_empleado.
  WRITE: / ls_empleado-id, ls_empleado-nombre.
ENDLOOP.
🛠️

Operaciones básicas con tablas

APPEND, READ, CLEAR, DELETE, SORT
InstrucciónQué haceEjemplo
APPEND Añade una fila al final de la tabla. APPEND ls_fila TO lt_tabla.
READ TABLE Busca una fila (primera coincidencia) y la copia a una estructura. Actualiza sy-subrc. READ TABLE lt_tabla INTO ls_fila WITH KEY id = 1.
CLEAR (estructura) Pone todos los campos de la estructura a su valor inicial. CLEAR ls_fila.
CLEAR (tabla) Elimina todas las filas de la tabla. CLEAR lt_tabla.
DELETE Elimina filas que cumplan una condición. DELETE lt_tabla WHERE nota < 5.
SORT Ordena la tabla por uno o varios campos. SORT lt_tabla BY nombre ASCENDING.
DESCRIBE TABLE … LINES Cuenta el número de filas. DESCRIBE TABLE lt_tabla LINES lv_total.

Field Symbols — Punteros a datos

<fs_> no guarda datos, apunta a ellos

Un field symbol es como un puntero en C o una referencia en Java. No contiene datos, sino que apunta a una zona de memoria donde están los datos. Se declara con FIELD-SYMBOLS y se usa con < >.

La principal ventaja es que cuando asignas un field symbol a una fila de una tabla (con ASSIGNING), cualquier cambio que hagas a través del field symbol modifica directamente la tabla, sin necesidad de hacer un MODIFY.

lt_tabla[1]
nombre='Ana', nota=8
←
<fs_fila>
apunta aquí
Uso de field symbol
FIELD-SYMBOLS: <fs_tarea> TYPE ty_tarea.

LOOP AT lt_tareas ASSIGNING <fs_tarea>.
  IF <fs_tarea>-nota < 5.
    <fs_tarea>-estado = 'SUSP'.   " Modifica lt_tareas directamente
  ENDIF.
ENDLOOP.
🧠
Los ángulos < > son obligatorios y forman parte del nombre. No es un adorno.
📊

GROUP BY en LOOP — Agrupar filas

Como el GROUP BY de SQL pero en memoria

Desde ABAP 7.40, puedes agrupar filas de una tabla interna directamente en un LOOP con la adición GROUP BY. SAP agrupa las filas según el valor de uno o varios campos y te permite procesar cada grupo.

zejercicio_13_sq_tabla.abap
LOOP AT gt_empleados INTO gs_empleado
     GROUP BY gs_empleado-depto
     ASSIGNING FIELD-SYMBOL(<grupo>).

  WRITE: / 'Departamento:', <grupo>.

  LOOP AT GROUP <grupo> ASSIGNING FIELD-SYMBOL(<miembro>).
    WRITE: / <miembro>-nombre.
  ENDLOOP.
ENDLOOP.

Aquí <grupo> contiene el valor del campo por el que agrupas (ej. 'IT', 'HR'). <miembro> es cada fila dentro de ese grupo.

MESSAGE — Tipos y comportamiento

Cómo comunicarte con el usuario

MESSAGE muestra un texto al usuario. El tipo del mensaje (TYPE) determina cómo se muestra y si el programa continúa o se detiene.

TYPE 'S' (Success): aparece en la barra de estado (verde). El programa continúa.
TYPE 'E' (Error): aparece en rojo. El programa se detiene y el usuario debe corregir algo en la pantalla de selección.
TYPE 'W' (Warning): aparece en amarillo. El programa continúa.
TYPE 'I' (Information): abre una ventana popup. El programa se pausa hasta que el usuario cierra la ventana.
TYPE 'A' (Abort): ventana roja de error. Normalmente termina el programa, pero con TRY/CATCH podemos controlarlo.
Ejemplos básicos
MESSAGE 'Producto añadido' TYPE 'S'.
MESSAGE 'Error: seleccione un valor' TYPE 'E'.
MESSAGE '¿Está seguro?' TYPE 'W'.
MESSAGE 'Proceso terminado' TYPE 'I'.
💡
DISPLAY LIKE: puedes hacer que un mensaje tipo 'S' se vea rojo (como error) pero sin bloquear: MESSAGE 'Aviso' TYPE 'S' DISPLAY LIKE 'E'.

🆕 Capturar TYPE 'A' con TRY/CATCH (evitar que termine)

Si envolvemos un MESSAGE TYPE 'A' en un TRY ... CATCH, podemos capturar la excepción y decidir qué hacer, por ejemplo, preguntar al usuario si realmente quiere salir.

Confirmación sin salir del programa
DATA: lv_confirm TYPE abap_bool.

TRY.
    MESSAGE '¿Seguro que quieres cerrar?' TYPE 'A'.
    " Si llegamos aquí, el usuario pulsó 'Finalizar' (Enter)
    lv_confirm = abap_true.
  CATCH cx_sy_abap_message.
    " Si pulsó Cancelar (ESC) o la X, capturamos la excepción
    lv_confirm = abap_false.
ENDTRY.

IF lv_confirm = abap_true.
  LEAVE PROGRAM.
ELSE.
  MESSAGE 'Continuamos...' TYPE 'S'.
ENDIF.

CALL FUNCTION — Llamar a módulos de función

Usar funciones predefinidas de SAP

Los módulos de función son como procedimientos almacenados en el servidor SAP. Hay miles ya creados. Para usarlos, escribes CALL FUNCTION 'NOMBRE' y le pasas los parámetros que necesite.

Los parámetros se agrupan en:

  • EXPORTING: datos que envías a la función.
  • IMPORTING: datos que la función te devuelve.
  • TABLES: tablas internas que pasas.
  • EXCEPTIONS: posibles errores.
VRM_SET_VALUES
CALL FUNCTION 'VRM_SET_VALUES'
  EXPORTING
    id     = 'P_LISTA'
    values = lt_valores.

PERFORM / FORM — Subrutinas propias

Dividir el código en trozos reutilizables

Para no tener todo el código en un solo bloque, puedes crear subrutinas con FORM y llamarlas con PERFORM. Las subrutinas van al final del programa.

Ejemplo
START-OF-SELECTION.
  PERFORM calcular_promedio USING lt_notas CHANGING lv_promedio.

FORM calcular_promedio USING lt_datos TYPE tt_notas
                       CHANGING cv_prom TYPE p.
  ...
ENDFORM.

USING son parámetros de entrada (solo lectura). CHANGING son parámetros que pueden modificarse y devolver el cambio al programa principal.

Strings y operaciones matemáticas

to_upper, to_lower, strlen, concatenar, MOD, DIV, **

Funciones de texto

  • to_upper( texto ) convierte a mayúsculas.
  • to_lower( texto ) convierte a minúsculas.
  • strlen( texto ) devuelve la longitud.
  • CONCATENATE a b INTO c SEPARATED BY space une textos.

Uso de funciones de texto dentro de clases

zejercicio_04_sq_dia7.abap
CLASS lcl_textos DEFINITION.
  PUBLIC SECTION.
    METHODS set_texto IMPORTING texto TYPE string.
    METHODS txt_to_mayus.
    METHODS txt_to_minus.
    METHODS txt_contar.
  PRIVATE SECTION.
    DATA gv_texto TYPE string.
ENDCLASS.

CLASS lcl_textos IMPLEMENTATION.
  METHOD set_texto.
    gv_texto = texto.
  ENDMETHOD.

  METHOD txt_to_mayus.
    WRITE: / to_upper( gv_texto ).
  ENDMETHOD.

  METHOD txt_to_minus.
    WRITE: / to_lower( gv_texto ).
  ENDMETHOD.

  METHOD txt_contar.
    WRITE: / |Longitud: { strlen( gv_texto ) }|.
  ENDMETHOD.
ENDCLASS.

Operadores matemáticos especiales

  • MOD: resto de la división (ej. 7 MOD 2 = 1).
  • DIV: división entera (ej. 7 DIV 2 = 3).
  • **: potencia (ej. 2 ** 3 = 8).
zejercicio_01_sq.abap
IF numero MOD 2 = 0.
  WRITE: 'Par'.
ENDIF.

resultado = base ** exponente.

WRITE — Mostrar datos en pantalla

La salida básica de ABAP

WRITE es la instrucción más sencilla para sacar datos por pantalla. Se usa en programas simples y para depurar.

  • WRITE texto. escribe en la línea actual.
  • WRITE: / texto. salta a una nueva línea antes de escribir.
  • WRITE: / 'Nombre:', ls_persona-nombre. varios elementos separados por comas.
  • WRITE: /10 'Texto'. empieza a escribir en la columna 10.
  • ULINE. dibuja una línea horizontal.
  • FORMAT COLOR COL_HEADING. cambia el color de la fila (para títulos).

Para programas serios con tablas de datos, se usa ALV en lugar de WRITE.

📊

ALV Clásico (REUSE_ALV_GRID_DISPLAY)

Mostrar tablas con todas las funcionalidades

ALV (ABAP List Viewer) es el componente estándar para mostrar tablas de forma profesional, con ordenación, filtros, exportación a Excel, etc.

La versión clásica usa el módulo de función REUSE_ALV_GRID_DISPLAY. Necesitas preparar:

  • Field catalog (it_fieldcat): describe cada columna (campo, título, alineación).
  • Layout (is_layout): configuración visual (zebra, optimización de ancho).
  • Tabla de datos (t_outtab): la tabla interna con los datos.
Ejemplo básico
DATA: lt_fieldcat TYPE slis_t_fieldcat_alv,
      ls_fieldcat TYPE slis_fieldcat_alv,
      ls_layout   TYPE slis_layout_alv.

" Definir columna ALUMNO
ls_fieldcat-fieldname = 'ALUMNO'.
ls_fieldcat-seltext_l = 'Nombre del Alumno'.
APPEND ls_fieldcat TO lt_fieldcat.

" Layout
ls_layout-zebra = 'X'.
ls_layout-colwidth_optimize = 'X'.

CALL FUNCTION 'REUSE_ALV_GRID_DISPLAY'
  EXPORTING
    is_layout   = ls_layout
    it_fieldcat = lt_fieldcat
  TABLES
    t_outtab    = lt_tareas.

ALV OO (cl_salv_table)

La forma moderna, más fácil y potente

En lugar de un módulo de función, usas la clase cl_salv_table. Con ella, no necesitas construir el fieldcat manualmente (SAP lo genera automáticamente).

Ejemplo mínimo
DATA: gr_alv     TYPE REF TO cl_salv_table,
      gr_func    TYPE REF TO cl_salv_functions,
      gr_columns TYPE REF TO cl_salv_columns_table,
      gr_column  TYPE REF TO cl_salv_column_table.

TRY.
    cl_salv_table=>factory(
      IMPORTING r_salv_table = gr_alv
      CHANGING  t_table      = lt_datos ).

    gr_func = gr_alv->get_functions( ).
    gr_func->set_all( abap_true ).   " Activa todos los botones

    gr_columns = gr_alv->get_columns( ).
    gr_columns->set_optimize( abap_true ).

    gr_alv->display( ).

  CATCH cx_salv_msg.
    MESSAGE 'Error' TYPE 'E'.
ENDTRY.

OOP en ABAP — Clases, objetos, TRY/CATCH

Conceptos básicos para usar las clases de SAP

La programación orientada a objetos en ABAP usa los mismos conceptos que Java o C++. Tú no creas tus propias clases todavía, pero sí usas las que SAP proporciona.

Referencias a objetos

Una variable que va a contener un objeto se declara con TYPE REF TO. Inicialmente está vacía (INITIAL).

Creación de objetos

Puedes crear un objeto con CREATE OBJECT o, como en el caso de ALV, con un método de fábrica (factory).

Llamada a métodos

  • objeto->metodo( ) para métodos de instancia.
  • clase=>metodo_estatico( ) para métodos de clase.

Manejo de errores con TRY/CATCH

Los métodos OOP lanzan excepciones cuando ocurre un error. Debes capturarlas con TRY ... CATCH ... ENDTRY.

Ejemplo con TRY/CATCH
TRY.
    cl_salv_table=>factory(
      IMPORTING r_salv_table = gr_alv
      CHANGING  t_table      = lt_datos ).
    gr_alv->display( ).
  CATCH cx_salv_msg.
    MESSAGE 'Error en ALV' TYPE 'E'.
ENDTRY.

🆕 Constructor (CONSTRUCTOR)

El constructor es un método especial que se ejecuta automáticamente cuando creas un objeto con CREATE OBJECT. Se usa para inicializar los atributos.

zejercicio_05_sq_dia7.abap
CLASS lcl_alumno DEFINITION.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING i_nombre TYPE string
                i_nota   TYPE p DECIMALS 2.
    ...
  PRIVATE SECTION.
    DATA: gv_nombre TYPE string,
          gv_nota   TYPE p DECIMALS 2.
ENDCLASS.

CLASS lcl_alumno IMPLEMENTATION.
  METHOD constructor.
    gv_nombre = i_nombre.
    gv_nota   = i_nota.
    " Aquí podríamos llamar a un método privado que calcule algo
  ENDMETHOD.
  ...
ENDCLASS.

Creación del objeto:

DATA go_alumno TYPE REF TO lcl_alumno.
CREATE OBJECT go_alumno
  EXPORTING
    i_nombre = 'Ana'
    i_nota   = 7.5.

Características:

  • Solo puede haber un constructor por clase.
  • No tiene RETURNING, solo IMPORTING y opcionalmente EXCEPTIONS.
  • Se ejecuta antes de que puedas usar cualquier otro método del objeto.
  • Si no declaras ningún constructor, ABAP proporciona uno vacío por defecto.

🆕 Métodos estáticos (CLASS-METHODS)

Los métodos estáticos pertenecen a la clase, no a los objetos. Se pueden llamar sin necesidad de crear un objeto. Se declaran con CLASS-METHODS y se llaman con =>.

zejercicio_01_sq_dia7.abap
CLASS lcl_calculadora DEFINITION.
  PUBLIC SECTION.
    CLASS-METHODS sumar
      IMPORTING i_val1 TYPE p
                i_val2 TYPE p
      RETURNING VALUE(r_result) TYPE p.
ENDCLASS.

CLASS lcl_calculadora IMPLEMENTATION.
  METHOD sumar.
    r_result = i_val1 + i_val2.
  ENDMETHOD.
ENDCLASS.

START-OF-SELECTION.
  DATA: lv_res TYPE p.
  lv_res = lcl_calculadora=>sumar( i_val1 = 2 i_val2 = 3 ).
  WRITE: / lv_res.

🆕 Atributos privados y getters

Los atributos se declaran en la sección PRIVATE SECTION para que solo los métodos de la clase puedan acceder a ellos. Si necesitas consultar su valor desde fuera, creas getters (métodos que devuelven el valor).

Getters en lcl_alumno
CLASS lcl_alumno DEFINITION.
  PUBLIC SECTION.
    METHODS get_nombre RETURNING VALUE(r_nombre) TYPE string.
    METHODS get_nota   RETURNING VALUE(r_nota)   TYPE p DECIMALS 2.
  PRIVATE SECTION.
    DATA: gv_nombre TYPE string,
          gv_nota   TYPE p DECIMALS 2.
ENDCLASS.

CLASS lcl_alumno IMPLEMENTATION.
  METHOD get_nombre.
    r_nombre = gv_nombre.
  ENDMETHOD.
  METHOD get_nota.
    r_nota = gv_nota.
  ENDMETHOD.
ENDCLASS.

Uso:

WRITE: / go_alumno->get_nombre( ), go_alumno->get_nota( ).

Diccionario de Datos (DÍA 8)

Transacciones SE11, SE14 y cómo crear tus propias tablas

El Diccionario de Datos de ABAP es donde se definen todos los objetos de datos persistentes: tablas, dominios, elementos de dato, estructuras, etc. Aquí guardamos los moldes que luego usaremos en nuestros programas.

1. RUTAS DE NAVEGACIÓN (TRANSACCIONES CLAVE)

  • [SE11] – ABAP Dictionary: El cuartel general. Aquí creamos:
    • Dominios (técnico: tipo, longitud)
    • Elementos de Datos (semántico: etiquetas, textos)
    • Tablas Transparentes (base de datos física)
    • Estructuras (solo en memoria, no persistentes)
  • [SE38] – ABAP Editor: Donde escribimos el código para mover los datos.
  • [SE14] – Database Utility: Para "arreglar" tablas cuando cambian físicamente y dan error de activación.

2. JERARQUÍA DE CREACIÓN (EL ORDEN IMPORTA)

Para crear un campo desde cero, el camino es de abajo hacia arriba:

A
DOMINIO (Domain)

Define el tipo técnico: data type (CHAR, DATS, INT4...), número de caracteres, y si tiene valores fijos (range).

B
ELEMENTO DE DATO (Data Element)

Asigna el dominio y añade las etiquetas (texto corto, medio, largo) que verá el usuario.

C
TABLA / ESTRUCTURA

Introduces el componente (nombre técnico del campo) y su tipo (el elemento de dato anterior).

3. CONFIGURACIÓN DE TABLAS (CAMPOS CRÍTICOS EN SE11)

Al crear una tabla (siempre con prefijo Z o Y), hay 3 paradas obligatorias antes de activar:

  • MANDT: Primer campo siempre. Tipo: MANDT. Marcado como Key. Es el cliente.
  • Clave (Key): Campos que identifican el registro (ej. IDAL, IDMAT).
  • Technical Settings (botón en la barra superior):
    • Data Class: APPL0 (datos maestros, que no crecen mucho).
    • Size Category: 0 (de 0 a pocos miles de registros).
    • Buffering: Buffering not allowed (por defecto para aprendizaje).
  • Enhancement Category (menú Extras → Enhancement Category): seleccionar "Can be enhanced (character-type or numeric)".

5. EL PUENTE AL CÓDIGO (SE38) – CONCEPTOS "LINEA" Y "TABLA"

Para manipular los datos en un Reporte, definimos:

Work Area y Tabla Interna
DATA: ls_alumno TYPE zsalumprc26sq.  " LINEA DE TRABAJO (Work Area) – una 'ficha' única
DATA: lt_alumnos TYPE TABLE OF zsalumprc26sq. " TABLA INTERNA – el 'archivador' de fichas

FLUJO DE DATOS ESTÁNDAR:

  1. CLEAR ls_alumno. → Limpia la ficha actual.
  2. ls_alumno-campo = 'Valor'. → Rellena la ficha.
  3. APPEND ls_alumno TO lt_alumnos. → Mete la ficha en el archivador.
  4. LOOP AT lt_alumnos INTO ls_alumno. → Saca fichas una a una para leerlas.

6. TIPS DE FORMATEO EN PANTALLA (WRITE)

  • WRITE: /10 'Texto'. → Empieza a escribir en la columna 10.
  • ULINE. → Dibuja una línea horizontal.
  • FORMAT COLOR COL_HEADING. → Cambia el color de la fila (ej: COL_HEADING para títulos).
  • lines( lt_tabla ) → Función para contar cuántas filas tiene una tabla interna.
Ejemplo de informe con datos de diccionario
REPORT zmostrar_alumnos.

DATA: ls_alumno TYPE zsalumprc26sq,
      lt_alumnos TYPE TABLE OF zsalumprc26sq.

START-OF-SELECTION.
  SELECT * FROM ztalumprc26sq INTO TABLE lt_alumnos.

  FORMAT COLOR COL_HEADING.
  WRITE: / 5 'ID', 15 'Nombre', 35 'Nota'.
  ULINE.

  LOOP AT lt_alumnos INTO ls_alumno.
    FORMAT COLOR OFF.
    WRITE: / 5 ls_alumno-id, 15 ls_alumno-nombre, 35 ls_alumno-nota.
  ENDLOOP.

  ULINE.
  WRITE: 'Total alumnos:', lines( lt_alumnos ).

Resumen Día 8: El Diccionario es donde viven los moldes de datos. Las tablas (ZT...) guardan información permanentemente. Las estructuras (ZS...) son solo para el programa. En el código usamos TYPE para referenciarlas y creamos tablas internas para trabajar con los datos.
