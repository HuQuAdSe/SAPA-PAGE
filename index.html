¿Qué es ABAP?

El lenguaje de SAP — cómo funciona y por qué existe

ABAP significa Advanced Business Application Programming. Es el lenguaje de programación propietario de SAP, creado en los años 80 y que sigue siendo el núcleo de todos los sistemas SAP ERP, S/4HANA y muchas soluciones de negocio empresariales.

Piénsalo así: SAP es como un enorme sistema de gestión empresarial (facturación, almacén, contabilidad, RRHH...) que viene con miles de funcionalidades prediseñadas. ABAP es la herramienta que te permite personalizar, ampliar y crear nuevas funcionalidades específicas para cada empresa.

Característica 1

Interpretado + Compilado

El código ABAP se compila en un bytecode que se ejecuta sobre el servidor de aplicaciones SAP (no en el ordenador del usuario). El usuario solo ve la interfaz gráfica.

Característica 2

Orientado a la BD

ABAP tiene instrucciones SQL propias (SELECT, INSERT, UPDATE, DELETE) que se conectan directamente a la base de datos del sistema SAP (Oracle, HANA, SQL Server...).

Característica 3

Basado en eventos

No se ejecuta de arriba a abajo como Python. El código está organizado en "eventos" que SAP lanza en momentos concretos (al cargar la pantalla, al ejecutar, al hacer clic...).

Característica 4

Multicliente y multiidioma

ABAP soporta entornos multimandante (cada empresa tiene su "cliente") y multiidioma. Los textos se almacenan en elementos de texto traducibles.

¿Cómo se desarrolla en ABAP?

Todo el desarrollo se hace dentro del propio SAP, a través de transacciones (códigos de acceso rápido). No necesitas instalar ningún IDE en tu ordenador; SAP GUI es el cliente que te conecta al servidor:

TransacciónPara qué sirve

SE11Diccionario de datos — Tablas, dominios, estructuras, tipos.
SE16NVisor de tablas — Para ver los datos de cualquier tabla del sistema.
SM30Mantenimiento de tablas — Interfaz gráfica para editar datos de una tabla.

💡
Todo comienza con la transacción SE38.

Cuando entras al SAP GUI, introduces el código de transacción en el campo de comandos (arriba a la izquierda). Si escribes SE38 y pulsas Enter, llegas al editor ABAP. Desde ahí creas tu programa, le das un nombre (siempre comenzando por Z o Y para personalizaciones), escribes el código y lo activas (botón del mechero o F3).

Estructura de un programa ABAP

Las partes que componen cualquier programa, y el orden correcto

Un programa ABAP en SAP siempre tiene una estructura bien definida. No puedes poner las cosas en cualquier orden. Veamos la estructura mínima de un Reporte (el tipo de programa más habitual):

Estructura completa de un reporte ABAP
"-----------------------------------------------------------------
" 1. DECLARACIÓN DEL PROGRAMA
"-----------------------------------------------------------------
REPORT zmi_primer_programa.

"-----------------------------------------------------------------
" 2. TIPOS PROPIOS (moldes de datos reutilizables)
"-----------------------------------------------------------------
TYPES: BEGIN OF ty_alumno,
         nombre TYPE string,
         nota   TYPE p DECIMALS 2,
       END OF ty_alumno.

"-----------------------------------------------------------------
" 3. DECLARACIONES DE DATOS (variables, tablas...)
"-----------------------------------------------------------------
DATA: lv_texto    TYPE string,
      ls_alumno   TYPE ty_alumno,
      lt_alumnos  TYPE TABLE OF ty_alumno,
      btn_calcular TYPE char10.

"-----------------------------------------------------------------
" 4. PANTALLA DE SELECCIÓN
"-----------------------------------------------------------------
PARAMETERS: p_nombre TYPE string OBLIGATORY.
SELECTION-SCREEN PUSHBUTTON /1(12) btn_calcular USER-COMMAND calc.

"-----------------------------------------------------------------
" 5. EVENTO: INITIALIZATION (se ejecuta UNA VEZ antes de la pantalla)
"-----------------------------------------------------------------
INITIALIZATION.
  btn_calcular = 'Calcular'.

"-----------------------------------------------------------------
" 6. EVENTO: AT SELECTION-SCREEN (se ejecuta al interactuar)
"-----------------------------------------------------------------
AT SELECTION-SCREEN.
  CASE sy-ucomm.
    WHEN 'CALC'.
      " lógica del botón calcular
  ENDCASE.

"-----------------------------------------------------------------
" 7. EVENTO: START-OF-SELECTION (la lógica principal — al pulsar F8)
"-----------------------------------------------------------------
START-OF-SELECTION.
  lv_texto = p_nombre.
  WRITE: / 'Hola,', lv_texto.

"-----------------------------------------------------------------
" 8. SUBRUTINAS (FORM ... ENDFORM) — al final del programa
"-----------------------------------------------------------------
FORM mostrar_resultado.
  WRITE: / 'Resultado calculado.'.
ENDFORM.
✅
El orden importa, pero no todo es obligatorio.

En un programa básico, lo mínimo es REPORT y START-OF-SELECTION. El resto de bloques solo existen si los necesitas. Sin embargo, cuando los uses, deben estar en el orden indicado arriba.

Convenciones de nombres (naming conventions)

Por qué los nombres siempre empiezan por una letra concreta

En ABAP existe una convención de nomenclatura muy estricta que te permite saber, solo mirando el nombre de una variable, qué tipo de objeto es y qué alcance tiene. Esto es fundamental para que el código sea legible y mantenible en proyectos grandes.

g_Global — existe en todo el programagv_total, gt_usuarios
i_Importing — parámetro de entrada de un método/FORMiv_precio
e_Exporting — parámetro de salidaev_resultado
r_Returning — valor de retornorv_suma
c_Changing — parámetro que se modificacv_contador

Todos los objetos propios (tablas, programas, SmartForms...) deben empezar por Z o Y. SAP reserva el resto del espacio de nombres para sus objetos estándar. Si no lo haces, podrías sobreescribir objetos de SAP en una actualización del sistema.

TYPES — Moldes de datos

Crear tus propias estructuras de datos reutilizables

Un TYPES es como crear un molde. Defines cómo debe ser una "ficha" de datos: qué campos tiene y de qué tipo es cada uno. Luego usas ese molde para crear variables concretas con DATA.

La diferencia entre TYPES y DATA es fundamental: TYPES define el molde (no ocupa memoria), mientras que DATA crea una variable concreta usando ese molde (sí ocupa memoria).

Tipos primitivos de ABAP

TipoDescripciónEjemplo de uso
stringTexto de longitud variable (el más flexible). Permite cualquier cantidad de caracteres.lv_nombre TYPE string
char(n)Texto de longitud fija. Siempre ocupa n posiciones (rellena con espacios). Más rápido que string.lv_dni TYPE char9
iEntero (integer). Para números sin decimales. Rango: -2.147.483.648 a 2.147.483.647.lv_cantidad TYPE i
p DECIMALS nNúmero empaquetado (packed). Para cálculos financieros con decimales. Altamente preciso.lv_precio TYPE p DECIMALS 2
dFecha en formato AAAAMMDD (8 caracteres). SAP maneja internamente comparaciones de fechas.lv_fecha TYPE d
tHora en formato HHMMSS (6 caracteres).lv_hora TYPE t
n(n)Numérico de longitud fija. Solo cifras, con ceros a la izquierda.lv_cod TYPE n LENGTH 5
x(n)Hexadecimal (datos binarios).lv_raw TYPE x LENGTH 4
abap_boolBooleano de ABAP. Valor abap_true ('X') o abap_false (' ').lv_ok TYPE abap_bool

Definir una estructura con BEGIN OF / END OF

Una estructura agrupa varios campos relacionados bajo un mismo nombre. Es como una "ficha" con múltiples columnas.

TYPES con estructura anidada
" Tipo básico (un sólo campo)
TYPES: ty_nombre TYPE string.

" Estructura: grupo de campos relacionados
TYPES: BEGIN OF ty_alumno,
         id     TYPE i,
         nombre TYPE string,
         nota   TYPE p DECIMALS 2,
         activo TYPE abap_bool,
       END OF ty_alumno.

" Tabla interna de ese tipo (una lista de fichas)
TYPES: ty_t_alumnos TYPE TABLE OF ty_alumno.
⚠️
Convención: prefijo ty_ para los TYPES.

Siempre que definas un tipo propio, usa el prefijo ty_ (o tty_ para tipos de tabla). Así, cuando ves ty_alumno, sabes inmediatamente que es una definición de tipo, no una variable.

DATA — Variables y constantes

Cómo declarar, inicializar y usar variables en ABAP

DATA es la instrucción para declarar variables. Usando el molde (TYPES) que hayamos definido antes, o directamente un tipo primitivo, creamos la variable que guardará datos reales.

Formas de declarar variables
" Variables escalares simples
DATA: lv_nombre  TYPE string,
      lv_precio  TYPE p DECIMALS 2,
      lv_activo  TYPE abap_bool,
      lv_fecha   TYPE d.

" Work Area (estructura): una sola "fila" de datos
DATA: ls_alumno TYPE ty_alumno.

" Tabla interna: un listado de "filas"
DATA: lt_alumnos TYPE TABLE OF ty_alumno.

" Con valor inicial
DATA: lv_max    TYPE i VALUE 100,
      lv_moneda TYPE char3 VALUE 'EUR'.

" Constante (no se puede cambiar después)
CONSTANTS: lc_iva TYPE p DECIMALS 4 VALUE '0.21'.

" Referencia a objeto (para OOP)
DATA: lo_alv TYPE REF TO cl_salv_table.

Acceso a componentes de una estructura

Para acceder a un campo concreto de una estructura (work area), usas el operador guión (-):

Asignación y lectura de campos
" Asignar valores campo a campo
ls_alumno-id     = 1.
ls_alumno-nombre = 'María García'.
ls_alumno-nota   = 8.5.
ls_alumno-activo = abap_true.

" Leer un campo
WRITE: / ls_alumno-nombre.

" Limpiar toda la estructura a valores vacíos
CLEAR ls_alumno.

" Copiar toda una estructura
DATA: ls_alumno2 LIKE ls_alumno.
ls_alumno2 = ls_alumno. " copia todos los campos
ℹ️
LIKE vs TYPE

Con TYPE ty_alumno defines el tipo usando el molde. Con LIKE ls_alumno defines el tipo copiando la estructura de otra variable existente. LIKE es útil cuando quieres asegurarte de que dos variables siempre tengan exactamente la misma estructura.

PARAMETERS — Campos de entrada en la pantalla

Los controles más básicos: campos de texto, checkboxes, etc.

PARAMETERS es la instrucción que añade un campo de entrada a la pantalla de selección. Cuando el usuario ejecuta el programa, ve esta pantalla y puede introducir valores. Es la forma más sencilla de recibir datos del usuario.

Variantes de PARAMETERS

Todos los tipos de PARAMETERS
" Campo de texto simple
PARAMETERS: p_nombre TYPE string.

" Campo obligatorio — el usuario NO puede dejarlo vacío
PARAMETERS: p_id TYPE i OBLIGATORY.

" Con valor por defecto
PARAMETERS: p_pais TYPE char3 DEFAULT 'ESP'.

" Checkbox — devuelve 'X' si está marcado, '' si no
PARAMETERS: p_activo AS CHECKBOX DEFAULT 'X'.

" Desplegable (listbox) — requiere VRM_SET_VALUES en INITIALIZATION
PARAMETERS: p_modo TYPE char10 AS LISTBOX VISIBLE LENGTH 30.

" Campo de fecha (abre el calendario)
PARAMETERS: p_fini TYPE d.

" Campo oculto del usuario pero con grupo para ocultar/mostrar
PARAMETERS: p_bon AS CHECKBOX MODIF ID fin.

Usar el valor en START-OF-SELECTION

Leer el valor del parámetro
START-OF-SELECTION.
  IF p_activo = 'X'.
    WRITE: / 'El usuario activó la opción'.
  ENDIF.
  WRITE: / 'Nombre introducido:', p_nombre.
  WRITE: / 'ID introducido:',     p_id.

SELECT-OPTIONS — Rangos de búsqueda

Filtros con valor único, rango y exclusiones

SELECT-OPTIONS es un control más potente que PARAMETERS. En lugar de un solo valor, permite introducir un rango complejo: desde-hasta, valores individuales, exclusiones, etc. El usuario puede hacer clic en el icono de rango para abrir un selector avanzado.

Internamente, un SELECT-OPTIONS es una tabla interna con cuatro columnas:

CampoTipoQué contiene
SIGNchar1'I' (Include = incluir) o 'E' (Exclude = excluir)
OPTIONchar2'EQ' (igual), 'BT' (entre), 'GT' (mayor que), etc.
LOWtipo del campoEl valor (o el límite inferior de un rango)
HIGHtipo del campoEl límite superior de un rango (vacío si es un valor único)
Declaración y uso de SELECT-OPTIONS
" Declaración: FOR indica la columna de referencia (define el tipo)
SELECT-OPTIONS: so_id     FOR ls_alumno-id,
                so_nombre FOR ls_alumno-nombre.

" Uso en SELECT: el WHERE IN filtra con todo el rango definido por el usuario
START-OF-SELECTION.
  SELECT * FROM ztalumnos
    INTO TABLE lt_alumnos
    WHERE id     IN so_id
      AND  nombre IN so_nombre.
💡
La magia de SELECT-OPTIONS con IN.

Cuando usas WHERE campo IN so_campo, SAP automáticamente traduce todas las condiciones del rango (incluidos los rangos BT, las exclusiones EQ, etc.) a SQL. No tienes que gestionar eso manualmente.

PUSHBUTTON — Botones personalizados

Añadir botones propios a la pantalla de selección

Por defecto, la pantalla de selección solo tiene los botones estándar de SAP: Ejecutar (F8), Cancelar, etc. Con SELECTION-SCREEN PUSHBUTTON puedes añadir tus propios botones con cualquier texto y lógica.

Cada botón tiene un USER-COMMAND, un código (hasta 20 caracteres) que SAP guarda en sy-ucomm cuando el usuario pulsa el botón. Así puedes identificar qué botón fue pulsado.

1
Declarar la variable que contendrá el texto del botón

La variable almacenará el texto visible que tendrá el botón. Debe ser de tipo char con la anchura del botón.

2
Declarar el botón en la pantalla de selección

Con SELECTION-SCREEN PUSHBUTTON. Indica la posición (fila/columna), la variable con el texto y el USER-COMMAND.

3
Asignar el texto visible en INITIALIZATION

En INITIALIZATION, asigna a esa variable el texto que quieres que muestre el botón (por ejemplo, 'Añadir').

4
Reaccionar en AT SELECTION-SCREEN

En AT SELECTION-SCREEN, comprueba el valor de sy-ucomm para saber qué botón se pulsó.

zejercicio_5_sq_botones.abap — Ejemplo completo
" Paso 1: declarar variables para los textos de los botones
DATA: btn_add TYPE char10,
      btn_fin TYPE char10.

" Paso 2: declarar los botones en la pantalla
"  /1(10) = fila nueva (/) columna 1, anchura 10
"  25(10) = misma fila, columna 25, anchura 10
SELECTION-SCREEN:
  PUSHBUTTON /1(10) btn_add USER-COMMAND add,
  PUSHBUTTON 25(10) btn_fin USER-COMMAND fin.

" Paso 3: asignar textos en INITIALIZATION
INITIALIZATION.
  btn_add = 'Añadir'.
  btn_fin = 'Finalizar'.

" Paso 4: detectar pulsaciones en AT SELECTION-SCREEN
AT SELECTION-SCREEN.
  CASE sy-ucomm.
    WHEN 'ADD'.
      " Lógica para añadir (añadir fila, calcular, etc.)
      PERFORM añadir_elemento.
    WHEN 'FIN'.
      " Lógica para finalizar
      PERFORM finalizar_pedido.
  ENDCASE.
🚨
USER-COMMAND siempre en MAYÚSCULAS en el WHEN.

SAP convierte automáticamente el USER-COMMAND a mayúsculas. Si en el WHEN pones 'Add' o 'add', nunca coincidirá. Pon siempre 'ADD' en mayúsculas.

LISTBOX — Desplegables dinámicos

Cómo llenar un combo box con valores de tu programa

Para crear un desplegable (combo box) en la pantalla de selección, necesitas dos cosas:

  1. Declarar un PARAMETERS con la adición AS LISTBOX VISIBLE LENGTH n.
  2. Llenar una tabla especial de tipo vrm_values y pasársela al módulo de función VRM_SET_VALUES en el evento INITIALIZATION.

La tabla vrm_values es un tipo de SAP que tiene dos campos: key (el valor interno que recibe la variable) y text (el texto visible para el usuario).

Flujo completo de LISTBOX — ejemplo con categorías
" 1. Declaración del parámetro como listbox
PARAMETERS: p_categ TYPE char10 AS LISTBOX VISIBLE LENGTH 40.

" 2. Variables para las opciones
DATA: lt_valores TYPE vrm_values,
      ls_valor   TYPE vrm_value.

" 3. En INITIALIZATION, llenamos las opciones
INITIALIZATION.
  CLEAR lt_valores.

  " Opción 1
  ls_valor-key  = 'BASICO'.
  ls_valor-text = 'Plan Básico (hasta 10 usuarios)'.
  APPEND ls_valor TO lt_valores.

  " Opción 2
  ls_valor-key  = 'PRO'.
  ls_valor-text = 'Plan Pro (hasta 100 usuarios)'.
  APPEND ls_valor TO lt_valores.

  " Opción 3
  ls_valor-key  = 'ENT'.
  ls_valor-text = 'Plan Enterprise (ilimitado)'.
  APPEND ls_valor TO lt_valores.

  " 4. Conectar la tabla con el campo (ID debe ir en MAYÚSCULAS)
  CALL FUNCTION 'VRM_SET_VALUES'
    EXPORTING
      id     = 'P_CATEG'    " nombre del PARAMETERS en MAYÚSCULAS
      values = lt_valores.

" 5. En START-OF-SELECTION, p_categ tendrá el key elegido
START-OF-SELECTION.
  CASE p_categ.
    WHEN 'BASICO'. WRITE: / 'Seleccionaste Plan Básico'.
    WHEN 'PRO'.   WRITE: / 'Seleccionaste Plan Pro'.
    WHEN 'ENT'.   WRITE: / 'Seleccionaste Plan Enterprise'.
  ENDCASE.
🚨
Error silencioso: parámetro id en minúsculas.

El parámetro id de VRM_SET_VALUES debe ir en mayúsculas. Si pones 'p_categ' en minúsculas, el desplegable aparecerá vacío sin ningún mensaje de error, lo que lo hace muy difícil de detectar.

RADIOBUTTON GROUP — Botones de opción exclusiva

Elegir una sola opción entre varias

Los radiobuttons permiten al usuario elegir una opción de un grupo. Todos los que compartan el mismo nombre de grupo son mutuamente excluyentes: solo uno puede estar marcado a la vez.

Cada radiobutton es una variable de tipo char1. Cuando está marcado, su valor es 'X'; cuando no, es un espacio vacío ' '.

Radiobuttons — declaración y uso completo
" Grupo 'mod' con 3 opciones. DEFAULT 'X' = seleccionado por defecto
PARAMETERS: rb_bas RADIOBUTTON GROUP mod DEFAULT 'X',
            rb_fin RADIOBUTTON GROUP mod,
            rb_com RADIOBUTTON GROUP mod.

" Usa Text Elements para los textos: Goto → Text Elements → Text Symbols
" (así se pueden traducir)

START-OF-SELECTION.
  IF rb_bas = 'X'.
    WRITE: / 'Modo Básico seleccionado'.
  ELSEIF rb_fin = 'X'.
    WRITE: / 'Modo Financiero seleccionado'.
  ELSE.
    WRITE: / 'Modo Completo seleccionado'.
  ENDIF.
⚡

USER-COMMAND en radiobuttons

Forzar la actualización de la pantalla al cambiar de opción

Cuando el usuario cambia de radiobutton, la pantalla no se redibuja automáticamente. Para que los cambios visuales (activar/desactivar campos) se vean al instante, necesitas que al pulsar el radiobutton se dispare un evento de pantalla. Eso se consigue añadiendo USER-COMMAND al primer radiobutton del grupo.

zejercicio_15_sq_dia6.abap
" USER-COMMAND 'click' en el primer radiobutton del grupo
PARAMETERS: rb_bas RADIOBUTTON GROUP mod USER-COMMAND click DEFAULT 'X',
            rb_fin RADIOBUTTON GROUP mod,
            rb_com RADIOBUTTON GROUP mod.

Con USER-COMMAND click, cada vez que el usuario selecciona un radiobutton del grupo, SAP ejecuta AT SELECTION-SCREEN con sy-ucomm = 'CLICK' y luego AT SELECTION-SCREEN OUTPUT. Esto nos permite reevaluar la lógica de activación de campos en tiempo real.

✅
Solo necesitas USER-COMMAND en el primer radiobutton del grupo.

El comando dispara un evento para todo el grupo, así que no es necesario repetirlo en cada botón de radio.

LOOP AT SCREEN — Pantalla dinámica

Activar, desactivar u ocultar campos según el contexto

La pantalla de selección se describe internamente mediante una tabla llamada SCREEN. Cada fila representa un elemento de la pantalla y tiene atributos modificables:

  • screen-input: 1 si el campo es editable, 0 si es de solo lectura.
  • screen-invisible: 1 si el campo está oculto, 0 si es visible.
  • screen-group1: etiqueta de modificación (hasta 3 caracteres) asignada al declarar el campo con MODIF ID.

Paso 1: Asignar grupos de modificación a los campos

Asignar grupos MODIF ID
PARAMETERS: p_noper TYPE string    MODIF ID bas,
            p_falta TYPE d         MODIF ID fec,
            p_bank  AS CHECKBOX    MODIF ID fin,
            p_bonus TYPE p DECIMALS 2 MODIF ID fin.

Paso 2: Modificar la tabla SCREEN en AT SELECTION-SCREEN OUTPUT

Lógica de activación dinámica
AT SELECTION-SCREEN OUTPUT.
  LOOP AT SCREEN.                       " recorremos todos los elementos de pantalla
    CASE screen-group1.                 " según el grupo MODIF ID...
      WHEN 'BAS'.                        " campos del grupo 'bas'
        IF rb_bas = 'X' OR rb_com = 'X'.
          screen-input = 1.             " activado y editable
        ELSE.
          screen-input = 0.             " solo lectura (gris)
        ENDIF.
      WHEN 'FEC'.                        " campos del grupo 'fec'
        IF rb_fin = 'X'.
          screen-input = 0.             " desactivado en modo financiero
        ELSE.
          screen-input = 1.
        ENDIF.
      WHEN 'FIN'.                        " campos del grupo 'fin'
        IF rb_fin = 'X'.
          screen-input     = 1.
          screen-invisible = 0.         " visible
        ELSE.
          screen-invisible = 1.         " oculto
        ENDIF.
    ENDCASE.
    MODIFY SCREEN.                      " ¡OBLIGATORIO! Sin esto, los cambios se pierden
  ENDLOOP.
🚨
No olvides MODIFY SCREEN.

Si no pones MODIFY SCREEN dentro del LOOP, todos los cambios que hiciste en la fila de SCREEN se descartan. SAP no te avisa de este error porque es sintácticamente válido.

Ciclo de vida del programa — Los eventos

Por qué el código no se ejecuta de arriba a abajo

En ABAP, el programa no se ejecuta en orden secuencial como en Python o JavaScript. En su lugar, escribes bloques de código asociados a eventos. SAP decide cuándo llamar a cada bloque según lo que haga el usuario.

Imagina que SAP es el director de orquesta: él decide cuándo suena cada sección. Tú compones la música de cada sección, pero no controlas el tempo.

Evento 1 — Al inicio

INITIALIZATION

Se ejecuta una sola vez, antes de que la pantalla de selección aparezca. Aquí preparas datos iniciales: cargar catálogos, asignar textos a botones, llenar listboxes, establecer valores por defecto.

Evento 2 — Al renderizar

AT SELECTION-SCREEN OUTPUT

Se ejecuta justo antes de que la pantalla se dibuje (al inicio y después de cada interacción que requiera redibujado). Aquí va la lógica para activar/desactivar/ocultar campos usando LOOP AT SCREEN.

Evento 3 — Al interactuar

AT SELECTION-SCREEN

Se ejecuta cada vez que el usuario interactúa con la pantalla: pulsa un botón, presiona Enter, hace clic en Ejecutar. Usa sy-ucomm para identificar qué acción ocurrió.

Evento 4 — Al ejecutar

START-OF-SELECTION

Se ejecuta cuando el usuario pulsa Ejecutar (F8) y la pantalla de selección ha pasado todas las validaciones. Aquí va la lógica principal: SELECT de la BD, cálculos, mostrar ALV, etc.

Orden de ejecución de los eventos
" ════════════════════════════════════════════
" ORDEN DE EJECUCIÓN (primera vez):
" 1. INITIALIZATION  →  preparar datos
" 2. AT SELECTION-SCREEN OUTPUT  →  configurar pantalla
" 3. [Usuario interactúa]
" 4. AT SELECTION-SCREEN  →  reaccionar a la interacción
" 5. AT SELECTION-SCREEN OUTPUT  →  reconfigurar pantalla
" 6. [Se repiten 3-5 hasta que el usuario pulsa F8]
" 7. AT SELECTION-SCREEN  →  con sy-ucomm = 'ONLI' (Ejecutar)
" 8. START-OF-SELECTION  →  la lógica principal
" ════════════════════════════════════════════
⚠️
Todo el código ejecutable debe estar dentro de un evento o FORM.

No puedes poner instrucciones "sueltas" fuera de los bloques de evento. Si lo haces, el activador de ABAP te dará un error de sintaxis. Los únicos elementos que pueden estar fuera de los eventos son las declaraciones (DATA, TYPES, CONSTANTS, PARAMETERS, SELECT-OPTIONS).

Las variables que empiezan por sy- son variables globales de sistema que SAP mantiene actualizadas en todo momento. No las declaras, ya existen. Son como los indicadores del coche: no puedes cambiarlos tú, pero te dicen la velocidad, el nivel de combustible, si hay algún fallo, etc.

" ❌ INCORRECTO: hacer otra cosa antes de comprobar

  " Falso positivo

CASE / WHEN — El switch de ABAP

Elegir entre múltiples opciones de forma limpia y eficiente

CASE evalúa una variable y ejecuta el bloque WHEN cuyo valor coincida. Es más legible que encadenar muchos ELSEIF cuando tienes múltiples opciones para una misma variable. Si ningún WHEN coincide, se ejecuta el bloque WHEN OTHERS.

CASE — Ejemplos completos
" CASE básico con sy-ucomm
AT SELECTION-SCREEN.
  CASE sy-ucomm.
    WHEN 'ADD'.
      PERFORM agregar_elemento.
    WHEN 'DEL'.
      PERFORM eliminar_elemento.
    WHEN 'CALC'.
      PERFORM calcular_total.
    WHEN OTHERS.
      " Otras acciones (por ejemplo, validación general)
  ENDCASE.

" CASE con una variable propia
CASE lv_estado.
  WHEN 'PEND'.  WRITE: / 'Pendiente'.
  WHEN 'PROC'.  WRITE: / 'En proceso'.
  WHEN 'DONE'.  WRITE: / 'Completado'.
  WHEN 'CANC'.  WRITE: / 'Cancelado'.
  WHEN OTHERS. WRITE: / 'Estado desconocido'.
ENDCASE.
💡
CASE vs IF — ¿cuándo usar cada uno?

Usa CASE cuando estás evaluando el mismo campo con diferentes valores posibles. Usa IF / ELSEIF cuando las condiciones son distintas o comparan diferentes variables. El CASE es más eficiente y más legible en el primer caso.

LOOP AT — Recorrer tablas internas

Iterar fila a fila sobre una tabla interna

LOOP AT es el bucle para recorrer tablas internas. En cada iteración, la instrucción INTO copia la fila actual en el work area (estructura) que indiques. La variable sy-tabix contiene el índice de la fila actual.

LOOP AT — Variantes principales
" LOOP AT básico: recorre TODAS las filas
LOOP AT lt_alumnos INTO ls_alumno.
  WRITE: / sy-tabix, ls_alumno-nombre, ls_alumno-nota.
ENDLOOP.

" LOOP AT con filtro WHERE: solo filas que cumplan la condición
LOOP AT lt_alumnos INTO ls_alumno WHERE nota >= 5.
  WRITE: / ls_alumno-nombre, 'APROBADO'.
ENDLOOP.

" LOOP AT con ASSIGNING: acceso directo a la fila (más eficiente)
LOOP AT lt_alumnos ASSIGNING <fs_alumno>.
  <fs_alumno>-nota = <fs_alumno>-nota + 1.   " modifica directamente en la tabla
ENDLOOP.

" LOOP AT con EXIT: salir del bucle al cumplir una condición
LOOP AT lt_alumnos INTO ls_alumno.
  IF ls_alumno-nota = 10.
    WRITE: / 'Primer 10 encontrado:', ls_alumno-nombre.
    EXIT.                    " sale del LOOP
  ENDIF.
ENDLOOP.

" LOOP AT con CONTINUE: saltar al siguiente ciclo
LOOP AT lt_alumnos INTO ls_alumno.
  IF ls_alumno-activo <> 'X'.
    CONTINUE.              " ignora alumnos inactivos
  ENDIF.
  WRITE: / ls_alumno-nombre.
ENDLOOP.
💡
INTO vs ASSIGNING — ¿cuál usar?

INTO ls_estructura copia la fila: si modificas ls_estructura, no modifica la tabla original (necesitas MODIFY después). ASSIGNING <fs> crea una referencia directa a la fila: cualquier cambio en <fs> se refleja inmediatamente en la tabla, sin necesidad de MODIFY. Usa ASSIGNING cuando vayas a modificar las filas.

DO n TIMES — Bucle con contador

Ejecutar un bloque un número determinado de veces

DO es el bucle contado de ABAP. Puedes indicar cuántas veces debe repetirse con DO n TIMES, o dejarlo indefinido (con cuidado de poner un EXIT para que no sea infinito). La variable sy-index lleva el contador (empieza en 1).

DO TIMES — Ejemplos
" Bucle contado: exactamente 5 veces
DO 5 TIMES.
  WRITE: / 'Iteración:', sy-index.
ENDDO.

" Bucle con variable como contador
DATA: lv_max TYPE i VALUE 10.
DO lv_max TIMES.
  ls_alumno-id   = sy-index.
  ls_alumno-nota = sy-index * 10.
  APPEND ls_alumno TO lt_alumnos.
ENDDO.

" Bucle indefinido con EXIT (equivalente a WHILE)
DO.
  IF sy-index > 100.
    EXIT.
  ENDIF.
  " ... lógica ...
ENDDO.

APPEND, READ, MODIFY, DELETE, CLEAR

Las operaciones CRUD sobre tablas internas

APPEND — Añadir una fila

APPEND
" Preparamos la fila (work area)
ls_alumno-id     = 1.
ls_alumno-nombre = 'Ana García'.
ls_alumno-nota   = 8.5.

" La añadimos al final de la tabla
APPEND ls_alumno TO lt_alumnos.

" Limpiar el work area para preparar la siguiente fila
CLEAR ls_alumno.

" Si la tabla ya contiene la misma estructura que la BD, podemos seleccionar
" directamente INTO TABLE para llenar la tabla de golpe:
SELECT * FROM ztalumnos INTO TABLE lt_alumnos.

READ TABLE — Buscar una fila

READ TABLE
" Buscar por valor de campo (búsqueda lineal)
READ TABLE lt_alumnos INTO ls_alumno WITH KEY id = 1.
IF sy-subrc = 0.
  WRITE: / 'Encontrado:', ls_alumno-nombre.
ELSE.
  WRITE: / 'No encontrado'.
ENDIF.

" Buscar por índice (número de fila)
READ TABLE lt_alumnos INTO ls_alumno INDEX 3.

" Buscar con ASSIGNING (referencia directa, más eficiente)
READ TABLE lt_alumnos ASSIGNING <fs_alumno> WITH KEY nombre = 'Ana García'.

MODIFY TABLE — Modificar una fila

MODIFY TABLE
" Tras un READ TABLE con INTO, modificamos el work area y escribimos de vuelta
READ TABLE lt_alumnos INTO ls_alumno WITH KEY id = 1.
IF sy-subrc = 0.
  ls_alumno-nota = 9.0.
  MODIFY lt_alumnos FROM ls_alumno INDEX sy-tabix. " sy-tabix = índice de la fila leída
ENDIF.

" Con ASSIGNING no necesitas MODIFY (se modifica directamente)
READ TABLE lt_alumnos ASSIGNING <fs_alumno> WITH KEY id = 1.
IF sy-subrc = 0.
  <fs_alumno>-nota = 9.0. " ya queda guardado en la tabla
ENDIF.

DELETE TABLE — Eliminar filas

DELETE TABLE
" Eliminar una fila específica por clave
DELETE lt_alumnos WHERE id = 1.

" Eliminar la fila actual dentro de un LOOP
LOOP AT lt_alumnos INTO ls_alumno.
  IF ls_alumno-nota < 5.
    DELETE lt_alumnos INDEX sy-tabix.
  ENDIF.
ENDLOOP.

" Vaciar toda la tabla
CLEAR  lt_alumnos.    " elimina todas las filas, mantiene la definición
FREE   lt_alumnos.    " libera también la memoria (útil con tablas grandes)

GROUP BY en LOOP — Agrupar filas

Recorrer una tabla agrupando por un campo (como SQL GROUP BY)

Desde ABAP 7.40, puedes usar LOOP AT ... GROUP BY para agrupar las filas de una tabla interna por el valor de un campo, similar al GROUP BY de SQL. Para cada grupo, SAP te da acceso a todas las filas del grupo y a datos del grupo (como el número de filas).

LOOP AT GROUP BY — agrupar alumnos por curso
TYPES: BEGIN OF ty_alumno_ext,
         nombre TYPE string,
         curso  TYPE string,
         nota   TYPE p DECIMALS 2,
       END OF ty_alumno_ext.

DATA: lt_alumnos TYPE TABLE OF ty_alumno_ext.

" LOOP con agrupación por curso
LOOP AT lt_alumnos INTO ls_alumno
  GROUP BY ls_alumno-curso
  ASCENDING
  ASSIGNING <fs_grupo>.

  WRITE: / 'CURSO:', <fs_grupo>-grupo-curso.

  " Sub-loop para iterar por las filas del grupo actual
  LOOP AT GROUP <fs_grupo> ASSIGNING <fs_alumno>.
    WRITE: / '  -', <fs_alumno>-nombre, <fs_alumno>-nota.
  ENDLOOP.

ENDLOOP.

TRY / CATCH para manejar errores graves

    " Cualquier otra excepción
    MESSAGE 'Error inesperado. Contacte con el administrador.' TYPE 'E'.
ENDTRY.

PERFORM y FORM — Ejemplo completo con parámetros
" Llamada a la subrutina (desde cualquier evento)

FORM/PERFORM vs Métodos de clase (OOP)

Las subrutinas FORM son el enfoque clásico y todavía muy usado en código legacy. Para código nuevo, se prefieren los métodos de clase (OOP) porque son más modulares, reutilizables y fáciles de testear. Sin embargo, FORM/PERFORM sigue siendo perfectamente válido para programas simples.

OOP en ABAP — Clases y objetos

Programación orientada a objetos para organizar código complejo

La Programación Orientada a Objetos (OOP) en ABAP usa los mismos conceptos que Java o C#. Una clase es un molde que define datos (atributos) y comportamientos (métodos). Un objeto es una instancia concreta de esa clase.

Estructura de una clase ABAP

Estructura completa de una clase local
"═══════ DEFINITION (qué tiene y puede hacer) ═══════
CLASS lcl_alumno DEFINITION.
  PUBLIC SECTION.                    " accesible desde fuera
    METHODS:
      constructor
        IMPORTING i_nombre TYPE string
                  i_nota   TYPE p DECIMALS 2,
      mostrar_datos,
      get_nombre RETURNING VALUE(r_val) TYPE string,
      get_nota   RETURNING VALUE(r_val) TYPE p DECIMALS 2.

  PRIVATE SECTION.                   " solo accesible desde dentro
    DATA:
      gv_nombre TYPE string,
      gv_nota   TYPE p DECIMALS 2,
      gv_estado TYPE string.        " 'APROBADO' o 'SUSPENSO'
    METHODS:
      calcular_estado.               " método privado (interno)
ENDCLASS.

"═══════ IMPLEMENTATION (cómo funciona) ═══════
CLASS lcl_alumno IMPLEMENTATION.

  METHOD constructor.
    gv_nombre = i_nombre.
    gv_nota   = i_nota.
    calcular_estado( ).            " se llama internamente
  ENDMETHOD.

  METHOD calcular_estado.          " privado — lógica interna
    IF gv_nota >= 5.
      gv_estado = 'APROBADO'.
    ELSE.
      gv_estado = 'SUSPENSO'.
    ENDIF.
  ENDMETHOD.

  METHOD mostrar_datos.
    WRITE: / gv_nombre, gv_nota, gv_estado.
  ENDMETHOD.

  METHOD get_nombre. r_val = gv_nombre. ENDMETHOD.
  METHOD get_nota.   r_val = gv_nota.   ENDMETHOD.

  DATA: lo_alumno TYPE REF TO lcl_alumno.

    EXPORTING
      i_nombre = 'Ana García'

Constructor — inicializar el objeto al crearlo

El CONSTRUCTOR es un método especial que se ejecuta automáticamente cuando creas un objeto con CREATE OBJECT. Solo puede tener parámetros IMPORTING (nunca RETURNING). Se usa para inicializar los atributos del objeto.

✅
Una sola instancia, múltiples objetos.

Puedes crear múltiples objetos de la misma clase: CREATE OBJECT lo_alumno1, CREATE OBJECT lo_alumno2... Cada uno tiene sus propios atributos independientes. La clase es el molde, los objetos son los ladrillos hechos con ese molde.

Métodos estáticos — CLASS-METHODS

Los métodos estáticos pertenecen a la clase, no a una instancia concreta. Se pueden llamar sin crear un objeto, usando el operador => (en lugar de ->).

Métodos estáticos — zejercicio_01_sq_dia7.abap
CLASS lcl_calc DEFINITION.
  PUBLIC SECTION.
    CLASS-METHODS:
      sumar
        IMPORTING i_a TYPE p DECIMALS 2
                  i_b TYPE p DECIMALS 2
        RETURNING VALUE(r_res) TYPE p DECIMALS 2,
      calcular_iva
        IMPORTING i_base TYPE p DECIMALS 2
        RETURNING VALUE(r_iva) TYPE p DECIMALS 2.
ENDCLASS.

CLASS lcl_calc IMPLEMENTATION.
  METHOD sumar.
    r_res = i_a + i_b.
  ENDMETHOD.
  METHOD calcular_iva.
    r_iva = i_base * '0.21'.
  ENDMETHOD.
ENDCLASS.

START-OF-SELECTION.
  DATA: lv_resultado TYPE p DECIMALS 2.
  " No necesita CREATE OBJECT — se llama directamente con =>
  lv_resultado = lcl_calc=>sumar( i_a = 100 i_b = 50 ).
  WRITE: / lv_resultado.   " 150
  WRITE: / lcl_calc=>calcular_iva( i_base = lv_resultado ). " 31.50

Atributos privados y Getters

Los atributos de la sección PRIVATE SECTION solo son accesibles desde dentro de la clase. Para que código externo pueda leer sus valores, se crean métodos getters (métodos que devuelven el valor del atributo).

Esto implementa el principio de encapsulación: el estado interno del objeto es privado, pero se expone de forma controlada a través de métodos públicos.

1. Jerarquía de creación — El orden importa

Para crear un campo de datos desde cero, el camino va de lo más específico a lo más general:

A
DOMINIO (Domain)

Define el tipo técnico del dato: tipo de dato SAP (CHAR, INT4, DATS, CURR, NUMC...), longitud, número de decimales y opcionalmente una lista de valores fijos permitidos (value range). Es el nivel más bajo de la jerarquía.

B
ELEMENTO DE DATO (Data Element)

Asigna un dominio a una etiqueta semántica. Añade los textos descriptivos que verá el usuario (texto corto, medio, largo, cabecera de columna). También puede vincular ayudas de búsqueda (F4).

C
TABLA o ESTRUCTURA

Agrupa varios campos (cada uno referenciando a un elemento de dato). Una tabla persiste en la base de datos; una estructura es solo un molde en memoria.

Work Area y Tabla Interna — El puente al código

Cómo se usan las tablas del diccionario en el código ABAP

Una vez creada la tabla en SE11, la usas en el código de dos formas: como work area (una sola fila, para leer/escribir fila a fila) y como tabla interna (un buffer en memoria con todas las filas).

Declaración, lectura y escritura de tablas del diccionario
"─── Declaración ───────────────────────────────────────
DATA: ls_alumno  TYPE          ztalumnos.       " Work Area (1 fila)
DATA: lt_alumnos TYPE TABLE OF ztalumnos.       " Tabla interna (N filas)

"─── Leer todos los registros de la BD → tabla interna ─
SELECT * FROM ztalumnos
  INTO TABLE lt_alumnos
  WHERE nota > 5.

"─── Leer un único registro de la BD → work area ───────
SELECT SINGLE * FROM ztalumnos
  INTO ls_alumno
  WHERE idalumno = '00001'.

"─── Insertar (crear registro nuevo en la BD) ──────────
ls_alumno-idalumno = '00010'.
ls_alumno-nombre   = 'Nuevo Alumno'.
ls_alumno-nota     = 7.
INSERT ztalumnos FROM ls_alumno.

"─── Actualizar (modificar registro existente en la BD) ─
ls_alumno-nota = 9.
MODIFY ztalumnos FROM ls_alumno.   " INSERT si no existe, UPDATE si existe
UPDATE ztalumnos SET nota = 9 WHERE idalumno = '00010'.

"─── Eliminar registro de la BD ────────────────────────
DELETE ztalumnos WHERE idalumno = '00010'.

"─── Hacer los cambios permanentes (commit) ────────────
COMMIT WORK.    " confirma todos los cambios en la BD
ROLLBACK WORK. " deshace todos los cambios no confirmados
🚨
INSERT vs UPDATE vs MODIFY — ¿cuál usar?

INSERT falla si ya existe el registro (retorna sy-subrc = 4). UPDATE falla si el registro NO existe. MODIFY funciona como un "upsert": inserta si no existe, actualiza si existe. En la mayoría de casos de negocio, MODIFY es la opción más segura.

SQL en ABAP — SELECT, INSERT, UPDATE, DELETE

Cómo interactuar con la base de datos desde el código

ABAP tiene su propio dialecto SQL, llamado Open SQL, que es independiente de la base de datos subyacente (Oracle, HANA, SQL Server). SAP traduce el Open SQL al SQL nativo de la BD automáticamente.

SELECT — Leer datos

Variantes de SELECT
"─── SELECT * INTO TABLE: leer todo en tabla interna ───
SELECT * FROM zttrprc26sq
  INTO TABLE lt_tareas
  WHERE estado IN so_estado
    AND responsable = sy-uname
  ORDER BY fechafin.
IF sy-subrc = 0.
  " Hay datos
ENDIF.

"─── SELECT SINGLE: leer exactamente un registro ────────
SELECT SINGLE titulo responsable nota
  FROM zttrprc26sq
  INTO (ls_tarea-titulo, ls_tarea-responsable, ls_tarea-nota)
  WHERE idtarea = '00001'.

"─── SELECT con DISTINCT y campos específicos ───────────
SELECT DISTINCT responsable
  FROM zttrprc26sq
  INTO TABLE lt_responsables.

"─── SELECT con JOIN ────────────────────────────────────
SELECT t~idtarea, t~titulo, u~nombre_usuario
  FROM zttrprc26sq AS t
  INNER JOIN usr21 AS u ON t~responsable = u~bname
  INTO TABLE @lt_result.

"─── SELECT con COUNT / SUM / MAX ────────────────────────
SELECT COUNT(*) FROM zttrprc26sq INTO lv_total.
SELECT MAX( nota ) FROM zttrprc26sq INTO lv_max_nota.

Escritura en la base de datos

SmartForms — ¿Qué son y para qué sirven?

La herramienta de SAP para diseñar formularios de impresión: PDFs, recibos, facturas, listas

SmartForms es la herramienta de SAP para crear formularios de impresión: facturas, albaranes, certificados, listados de alumnos, recibos de nómina, etc. Se diseñan visualmente en la transacción SMARTFORMS y se invocan desde un programa ABAP.

El flujo de trabajo completo tiene 6 pasos:

Paso 1 — SE11

Tipo de tabla

Crear el Table Type (ZTT_...) basado en la tabla de BD. SmartForms solo acepta Table Types, no tablas directamente.

Paso 2 — SMARTSTYLES

Estilos

Crear al menos 3 estilos tipográficos: título, cabecera de tabla y datos. El formulario los usará para dar formato al texto.

Paso 3 — SMARTFORMS

Configuración

Definir la interfaz (parámetros de entrada) y las variables globales. Vincular el estilo al formulario.

Paso 4 — SMARTFORMS

Layout

Diseñar visualmente el formulario: tamaño de página, logo, ventana de título y tabla de datos.

Paso 5 — SMARTFORMS

Celdas y variables

Añadir variables ABAP en las celdas: &WA_ALUMNO-NOMBRE&, &WA_ALUMNO-NOTA&, etc.

Paso 6 — SE38

Programa de llamada

Escribir el programa ABAP que recupera los datos y llama al SmartForm para generar el PDF.

⚠️
El orden es estricto.

Debes crear los objetos en el orden indicado. Si intentas usar un Table Type que no existe, o un estilo sin activar, el SmartForm dará errores en tiempo de activación. Activa cada objeto antes de pasar al siguiente.

1️⃣

Paso 1: Tipo de tabla en SE11

¿Por qué SmartForms necesita un Table Type y no la tabla directamente?

SmartForms no acepta tablas de base de datos directamente en la pestaña "Tables" de la interfaz. Necesitas un objeto intermedio llamado Table Type (tipo de tabla), que es básicamente una definición de "tabla interna de este tipo de fila".

Un Table Type se crea en SE11 y tiene dos componentes clave:

  • Nombre: por convención ZTT_NOMBRE (con doble T para Table Type).
  • Line Type: el nombre de la tabla de base de datos cuya estructura define cada fila.
1
Abrir SE11 y crear el objeto

En la transacción SE11, selecciona la opción "Data Type" (no "Database Table"), escribe el nombre ZTT_ALUMNOS_SQ y pulsa Create.

2
Seleccionar "Table Type"

En el diálogo que aparece, elige "Table Type". Esto indica que vas a definir el tipo de una tabla interna (una lista de filas del tipo indicado).

3
Indicar el Line Type

En el campo "Line Type", escribe el nombre de tu tabla de base de datos: ZTALUMPRC26SQ. Esto le dice a SAP que cada fila de la tabla interna tiene la misma estructura que la tabla de BD.

4
Guardar y ACTIVAR

Pulsa Save y luego el botón de activar (icono del fósforo). Sin activar no sirve de nada — el SmartForm dará un error al validar la interfaz.

🚨
Error más frecuente: olvidar activar el Table Type.

Si en SmartForms introduces el Table Type y SAP no lo reconoce o da error al comprobar la sintaxis, lo primero que debes verificar es que el Table Type está activo en SE11. Un objeto guardado pero no activo no es visible para el sistema.

2️⃣

Paso 2: Estilos con SMARTSTYLES

Definir la tipografía del formulario — mínimo 3 estilos

Los estilos en SmartForms son equivalentes a los estilos de Word o CSS: definen la fuente, el tamaño y el formato del texto. Antes de diseñar el formulario, debes crear al menos 3 estilos de párrafo: uno para el título, uno para las cabeceras de tabla y uno para los datos.

1
Entrar en SMARTSTYLES y crear el estilo

Transacción SMARTSTYLES → introduce el nombre ZST_ALUMNOS → pulsa Create (o Display si ya existe para editarlo).

2
Configurar el párrafo estándar en "Header Data"

En la carpeta "Header Data", en el campo "Standard Paragraph", escribe D1. Esto indica que el párrafo por defecto será el estilo de datos (el que usaremos para la mayoría del texto).

3
Crear los 3 formatos de párrafo

Haz clic derecho en "Paragraph Formats" → Create. Crea los tres estilos:

  • T1 → Pestaña Font: Fuente Arial, Tamaño 20pt, Negrita activada. Para el título del formulario.
  • C1 → Pestaña Font: Fuente Arial, Tamaño 12pt, Negrita activada. Para las cabeceras de la tabla.
  • D1 → Pestaña Font: Fuente Arial, Tamaño 10pt, Normal. Para los datos de la tabla.
4
Guardar y ACTIVAR el estilo

Igual que con el Table Type: guarda y activa. Sin activar, el SmartForm no podrá usar el estilo.

ℹ️
Los estilos se asignan en el editor de texto.

Dentro del SmartForm, cuando edites el texto de una celda, verás un menú desplegable de estilos (donde por defecto aparece '*'). Ahí puedes cambiar a T1 para el título, C1 para cabeceras, etc. Si el desplegable está vacío, es que el estilo no está asignado al formulario (ver Paso 3).

3️⃣

Paso 3: Configuración del SmartForm

Interfaz, variables globales y estilo

Ahora creas el formulario en sí (transacción SMARTFORMS) y lo configuras con tres elementos esenciales: la interfaz (qué datos recibe), las variables globales (qué variables usa internamente) y el estilo que aplicará.

1
Crear el formulario

En SMARTFORMS, introduce el nombre ZSF_ALUMNOS_PRC_SQ y pulsa Create. Aparecerá el editor visual con el árbol de nodos a la izquierda y el área de diseño a la derecha.

2
Definir la interfaz (Form Interface → Tables)

En el árbol de la izquierda, ve a Form Interface → subpestaña "Tables". Añade un parámetro de tabla:

  • Parameter Name: T_ALUMNOS
  • Type: TYPE
  • Associated Type: ZTT_ALUMNOS_SQ (el Table Type que creaste en el Paso 1)

Esto le dice al SmartForm: "voy a recibir una tabla de alumnos de este tipo desde el programa ABAP".

3
Definir la variable de trabajo (Global Definitions → Global Data)

En Global Definitions → subpestaña "Global Data", añade una variable:

  • Field Name: WA_ALUMNO
  • Type: TYPE
  • Associated Type: ZTALUMPRC26SQ (la tabla de BD — esta es la work area para una fila)

Esta variable recibirá fila a fila los datos de T_ALUMNOS durante el LOOP interno del SmartForm.

4
Asignar el estilo al formulario (Form Attributes)

En Form Attributes → subpestaña "Output Options", en el campo "Style", escribe ZST_ALUMNOS. Así el formulario usará los estilos tipográficos que creaste.

⚠️
La diferencia entre WA_ALUMNO y T_ALUMNOS.

T_ALUMNOS es la tabla interna completa (N filas). WA_ALUMNO es el work area (1 fila). En el SmartForm, la tabla recorre automáticamente T_ALUMNOS y en cada iteración coloca la fila actual en WA_ALUMNO. Así puedes referenciar &WA_ALUMNO-NOMBRE& en las celdas.

4️⃣

Paso 4: Diseño del layout

Logo, título y tabla de datos en la página

El layout es el diseño visual del formulario. Debes configurar el tamaño de página, añadir los elementos visuales (logo, título) y crear la tabla de datos en el área principal (MAIN).

1
Configurar la página (%PAGE1)

Haz clic en el nodo %PAGE1 en el árbol. En las propiedades:

  • Page Format: A4
  • Orientation: Portrait (vertical)
2
Añadir el logo (Graphic)

Haz clic derecho en el nodo de la página (%PAGE1) → Create → Graphic. En las propiedades del nodo gráfico:

  • Selecciona el objeto gráfico (logo cargado previamente en SE78).
  • En "Output Options": pon Left Margin en 16 CM para ubicarlo en la esquina superior derecha.
3
Añadir el título (Secondary Window)

Haz clic derecho en la página → Create → Secondary Window. Dentro de esa ventana secundaria:

  • Haz clic derecho → Create → Text.
  • En el editor de texto, escribe "LISTADO DE ALUMNOS".
  • Selecciona el texto y cambia el formato del párrafo de * a T1 (el estilo de título que creaste).
4
Crear la tabla de datos (en MAIN)

Haz clic derecho en el nodo MAIN → Create → Table. En las propiedades del nodo Table:

  • Pestaña "Table": Si no ves las columnas automáticamente, dibújalas manualmente con el Table Painter (herramienta visual de columnas).
  • Pestaña "Data": En "Internal Table" escribe T_ALUMNOS; en "Into" escribe WA_ALUMNO.
5️⃣

Paso 5: Celdas y variables

Añadir cabeceras y datos con variables &WA_ALUMNO-CAMPO&

Dentro del nodo Table hay dos secciones: Header (cabecera que se repite en cada página) y Main Area (donde van los datos de cada fila). En cada sección se crean "Table Lines" con celdas que contienen texto o variables.

1
Crear la cabecera de la tabla (Header)

Haz clic derecho en Header → Create → Table Line. En cada celda de esa línea, haz clic derecho → Create → Text y escribe el texto de la cabecera:

  • Celda 1: NOMBRE → Formato: C1
  • Celda 2: APELLIDO 1 → Formato: C1
  • Celda 3: APELLIDO 2 → Formato: C1
  • Celda 4: EMAIL → Formato: C1
2
Crear las filas de datos (Main Area)

Haz clic derecho en Main Area → Create → Table Line. En cada celda, inserta el texto con la variable del campo correspondiente del work area:

  • Celda 1: &WA_ALUMNO-NOMB& → Formato: D1
  • Celda 2: &WA_ALUMNO-APE1& → Formato: D1
  • Celda 3: &WA_ALUMNO-APE2& → Formato: D1
  • Celda 4: &WA_ALUMNO-EMAIL& → Formato: D1

Las variables van entre ampersands (&...&). SmartForms las reemplaza automáticamente en tiempo de impresión por el valor real del campo.

3
Guardar y ACTIVAR el SmartForm

Guarda con Ctrl+S y activa con el botón del fósforo (o Ctrl+F3). Si hay errores de sintaxis o referencias rotas, aparecerán en el log de activación. Resuélvelos antes de continuar.

✅
Tip: Para ver la vista previa del diseño.

Una vez activo, puedes ejecutar el SmartForm directamente desde la transacción SMARTFORMS pulsando Form → Test. Te pedirá los parámetros de la interfaz (T_ALUMNOS en este caso). Si la interfaz los requiere, tendrás que llamarlo desde el programa ABAP.

6️⃣

Paso 6: Programa ABAP de llamada

El código que recupera los datos y lanza el SmartForm

SmartForms no se llaman directamente con el nombre del formulario. SAP genera internamente un módulo de función con el mismo nombre que el SmartForm, y ese módulo debe obtenerse dinámicamente con SSF_FUNCTION_MODULE_NAME. Esto permite cambiar el nombre del formulario sin tocar el código.

ZEJERCICIO_01_SQ_DIA14.abap — Programa completo de llamada a SmartForm
REPORT zejercicio_01_sq_dia14.

"═══ DATOS ════════════════════════════════════════════════════
DATA:
  lt_alumnos TYPE ztt_alumnos_sq,      " ← mismo tipo que el Table Type del SmartForm
  lv_fm_name TYPE rs38l_fnam,           " nombre dinámico del módulo de función
  ls_control TYPE ssfctrlop,             " opciones de control (preview, no_dialog...)
  ls_output  TYPE ssfcompop.             " opciones de salida (impresora, destino...)

"═══ PANTALLA (opcional) ══════════════════════════════════════
SELECT-OPTIONS: so_id FOR ls_alumno-idalumno.

"═══ LÓGICA PRINCIPAL ════════════════════════════════════════
START-OF-SELECTION.

  " 1. Recuperar los datos de la BD en la tabla interna
  SELECT nomb, ape1, ape2, email
    FROM ztalumprc26sq
    INTO CORRESPONDING FIELDS OF TABLE @lt_alumnos
    WHERE idalumno IN so_id.

  IF sy-subrc <> 0.
    MESSAGE 'No se encontraron alumnos con los criterios indicados.' TYPE 'W'.
    RETURN.
  ENDIF.

  " 2. Obtener el nombre del módulo de función generado por el SmartForm
  CALL FUNCTION 'SSF_FUNCTION_MODULE_NAME'
    EXPORTING
      formname = 'ZSF_ALUMNOS_PRC_SQ'   " nombre exacto del SmartForm
    IMPORTING
      fm_name  = lv_fm_name              " nombre dinámico resultado
    EXCEPTIONS
      no_form   = 1
      no_function_module = 2
      OTHERS   = 3.

  IF sy-subrc <> 0.
    MESSAGE 'SmartForm ZSF_ALUMNOS_PRC_SQ no encontrado o no activo.' TYPE 'E'.
    RETURN.
  ENDIF.

  " 3. Configurar las opciones de salida
  ls_control-no_dialog = 'X'.     " sin diálogo de impresión
  ls_control-preview   = 'X'.     " mostrar vista previa en pantalla
  ls_output-tddest     = 'LP01'. " impresora (si no hay preview)

  " 4. Llamar al SmartForm (mediante el módulo de función dinámico)
  CALL FUNCTION lv_fm_name
    EXPORTING
      control_parameters = ls_control
      output_options     = ls_output
      user_settings      = ' '     " ignorar settings del usuario
    TABLES
      t_alumnos          = lt_alumnos   " ← el parámetro definido en la interfaz
    EXCEPTIONS
      formatting_error   = 1
      internal_error     = 2
      send_error         = 3
      user_canceled      = 4
      OTHERS             = 5.

  IF sy-subrc <> 0.
    MESSAGE 'Error al generar el formulario PDF.' TYPE 'E'.
  ELSE.
    MESSAGE 'Formulario generado correctamente.' TYPE 'S'.
  ENDIF.

Errores frecuentes en SmartForms y cómo resolverlos

Los problemas más comunes y sus soluciones definitivas
🚨
Error 1: "Table type ZTT_ALUMNOS_SQ not found" en la interfaz.

Causa: El Table Type no está activado en SE11.
Solución: Ir a SE11, abrir ZTT_ALUMNOS_SQ, activar (botón del fósforo). Verificar que el estado dice "Active".

🚨
Error 2: El desplegable de estilos en el editor de texto está vacío o muestra solo '*'.

Causa: El estilo ZST_ALUMNOS no está asignado al formulario, o no está activo.
Solución 1: Verificar que ZST_ALUMNOS está activo en SMARTSTYLES.
Solución 2: En el formulario, ir a Form Attributes → Output Options → campo "Style" → escribir ZST_ALUMNOS.

🚨
Error 3: "EXCEPTION: no_form" al llamar SSF_FUNCTION_MODULE_NAME.

Causa: El nombre del formulario está mal escrito, o el SmartForm no está activo.
Solución: Verificar el nombre exacto (distingue mayúsculas/minúsculas). Ir a SMARTFORMS, abrir el formulario y verificar que está activo. Si dice "Inactive", activarlo.

⚠️
Error 4: La tabla aparece en blanco (sin datos) en el PDF.

Causa A: El SELECT del programa no devuelve datos (comprobar sy-subrc).
Causa B: En la pestaña "Data" del nodo Table, el nombre de la tabla interna o el work area están mal escritos (distingue mayúsculas).
Causa C: El Table Type en la interfaz y el tipo de la tabla interna en el programa no coinciden.

⚠️
Error 5: Las variables &WA_ALUMNO-CAMPO& aparecen literalmente en el PDF (no se sustituyen).

Causa: La variable WA_ALUMNO no está declarada en Global Definitions del SmartForm, o el nombre del campo está mal escrito.
Solución: Verificar la declaración en Global Definitions. El nombre debe coincidir exactamente con el de la tabla de BD (en mayúsculas).

⚠️
Error 6: La tabla no tiene columnas visibles en el Table Painter.

Causa: El Table Painter necesita que las columnas se dibujen manualmente la primera vez.
Solución: Dentro del nodo Table, busca el botón "Table Painter". Aparece una cuadrícula donde debes arrastrar para crear las columnas. Luego en cada columna creada, puedes asignar el ancho exacto.

Resumen ejecutivo
Regla 1

Activa todo

Cada objeto del Diccionario (Table Type, Domain, Data Element, Tabla) y cada SmartStyle debe estar activo antes de usarlo. Un objeto guardado pero no activo es invisible para el sistema.

Regla 2

Table Type, nunca la tabla directamente

SmartForms solo acepta Table Types en la interfaz. La tabla de BD se usa como Line Type del Table Type, no directamente.

Regla 3

SSF_FUNCTION_MODULE_NAME siempre

Nunca intentes llamar directamente al módulo de función generado por el SmartForm con su nombre literal. Siempre usa SSF_FUNCTION_MODULE_NAME para obtenerlo dinámicamente.

Regla 4

Tipo de datos coherente

El tipo de la tabla interna en el programa ABAP (TYPE ZTT_ALUMNOS_SQ) debe ser exactamente el mismo que el definido en la interfaz del SmartForm.
